*Concurrency is considered difficult for a few interconnected reasons:*
1. *Shared mutable state is the root of most problems. When multiple threads or processes can read and write the same data simultaneously, you get race conditions — bugs where the outcome depends on the unpredictable timing of operations. These bugs are notoriously hard to reproduce.*
	- problem: share mutable state
	- solution: don't share mutable state, allow mutation, but don't share

2. *Non-determinism makes debugging brutal. A concurrent bug might appear once every thousand runs, vanish under a debugger (because the debugger changes timing), and behave differently across machines or OS loads. You can’t reliably reproduce the conditions that caused the failure.*
	- problem: brutal debugging of determinism using sequential debugging methods
	- solution: build debuggers that are not based on the sequential mindset
		- hardware solved this problem many decades ago - used probes and oscilloscopes to view data flow in many places at once ; "dual trace" oscilloscopes allowed viewing two event flows at once ; hardware cost - the cost of oscilloscopes - was the problem in the 20th century, it is no longer the problem
		- in software "fan-out" allows us to insert (and remove) probes in software circuits, the synchronous mindset tends to discourage this idea because of the 1-in/1-out nature of "functions" and the idea that software units must return values instead of performing side-effects like displaying traces
			- the "fix" is easy - create software Parts (components) that don't call each other and don't cause blocking, by using queues instead of function calls

3. *Synchronization primitives are easy to misuse. Locks, semaphores, and mutexes solve race conditions but introduce new problems:*
	- problem: synchronization primitives are too low level, causing programmers to deal with them in a GOTO-like manner
	- solution: elide synchronization primitives with higher-level, architectural, explicit synchronization, e.g. use simple queue classes
		- queue insertion and extraction does need to be atomic, but doesn't happen often and uses only a small slice of time

4. *•	Deadlock — two threads each hold a lock the other needs, so both wait forever*
	- deadlock is an architectural issue, allow architects to deal with it explicitly instead of trying to find general solutions
	- deadlock is a per-project, high-level problem ; don't try to hide it, allow software architects to deal with the problems explicitly ; any "generalized" solution is slanted towards the biases of the implementor(s) and usually doesn't address the real needs of a particular project

5. *•	Livelock — threads keep reacting to each other without making progress*
	- same as *deadlock* - don't try to hide the problems, allow software architects to deal with the problems explicitly on a per-project basis

6. *•	Lock contention — too much locking kills the performance gains you wanted from concurrency in the first place*
	- problem: too much locking, too low level
	- solution: don't lock so much, don't share data that must be locked, use message sending to send only the data that matters - receiver Parts can employ mutation and state internally without sharing their state in a fine-grained manner
	
7. *The human mental model is sequential. Our brains naturally think step-by-step. Reasoning about multiple interleaved execution paths simultaneously — and all the ways they can interact — is genuinely cognitively hard. The number of possible execution orderings grows factorially with the number of operations.*
	- actually, our brains are capable of orchestrating the coordinated action of some 500+ moving parts, concurrently and in parallel
	- our "conscious" - "Self 1" - part of the brain can only deal with only one sequential problem at a time
	- solution: allow asynchrony 

8. *Testing is incomplete by nature. Unit tests run in a specific order. Even good test suites rarely explore the full space of thread interleavings, so concurrent bugs slip through and only surface in production under real load.*
	- problem: Testing is incomplete by nature, but, testing is still necessary, even after 50+ years of research
	- solution: test small, testable units, then compose larger systems using only tested Parts, and test the larger systems to verify that the compositions are reasonable
		- testing needs to be hierarchical - small Parts first, then small compositions of parts, then bigger compositions of compositions of parts, and so on.
