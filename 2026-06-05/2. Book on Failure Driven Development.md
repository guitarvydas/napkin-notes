I've begun writing new Leanpub book. In this book, I'm trying to give a recipe for _how_ to concretely implement ideas for thinking beyond the synchronous, sequential paradigm, knowing that just doing it will implicitly broaden one's outlook. Unfortunately, I don't think that this can be boiled down to a single, straight-line recipe and must consist of a mish-mosh of different ideas and techniques that can be applied at designers' discretion.

The techniques in include
- PBP - Parts Based Programming
- T2T - text to text transmogrification (convert a custom syntax to runnable code in an existing language)
- existing command line tools (awk, sed, jq, etc.)
- existing languages for specific programming issues, e.g. Prolog for exhaustive search
- how to stop adding bandaids to 'make' and do something simpler that allows for building and using tools that consist of combinations of the above technologies. For example, if you build a state-machine diagram to Python transmogrifier that operates like an enhanced pipeline using many tools, you should be able to use it to draw little state machines in other projects

One of the most concrete things I would like to demonstrate is the idea I learned in the few years that I studied Physics - invent custom notations and "spherical cows" to help you focus on specific aspects of a problem. This is exactly the opposite of what the programming community is doing now, i.e. we keep inventing general purpose programming languages that encourage us to ignore problems that don't fit neatly into the general purpose mold determined by the language designer(s) and encourage us to follow edicts that make sense only within the narrow paradigm chosen by the language, e.g. like threading every problem through the eye of the needle of synchronous, sequential thinking, using function-based languages like C, Python, Haskell, Clojure, Rust, etc. and claiming that "state is bad" - whereas state is only bad when used in an ad-hoc manner in a language (typically "functional") that doesn't support reasoning about state.

Humanity has been here many times before 
- Ptolemaic cosmology, overturned by Gallileo
- transistor circuits -> programming languages
	- we built devices using fixed-capability ICs and electronic components until someone invented the CPU IC - a simple sequencer exposing only a narrow waist to the capabilities of electronics - which led to the creation of the field of Computer Science
- Nobel laureate Ilya Prigogene's assertion that functional thinking has set physics back by 100 years
- physicist Richard Feynman escaped from the confines of mathematical notation and invented Feynman Diagrams
- tell: "dark matter" is a tacit admission that we don't understand much of cosmology and that functional notation isn't making our task much easier

incorrect belief: mathematics is _the_ universal language, hence, mathematics is a natural fit for reprogrammable machines ("computers")

Programming Languages are just _subsets_ of a programming toolchain, we need to compose solutions using many PLs, not just one GPL at a time. Using many approaches and ideas.
