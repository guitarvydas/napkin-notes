**L;R**

**What the book is about**

The book is a practical guide to thinking and building software beyond the synchronous, sequential paradigm.

It is not a single straight-line recipe. It is a collection of techniques and ideas designers can combine at their discretion. The core techniques include:

Parts-Based Programming (PBP). Text-to-text transmogrification (T2T) — converting custom syntax to runnable code in an existing language. Existing command-line tools such as `awk`, `sed`, and `jq`. Existing languages chosen for specific problems, such as Prolog for exhaustive search. A simpler alternative to `make` that lets you build and reuse tool pipelines — for example, a state-machine diagram-to-Python transmogrifier you can drop into other projects.

The most concrete idea I want to demonstrate is one I learned from studying Physics: invent custom notations and "spherical cows" to help you focus on specific aspects of a problem. This is precisely the opposite of what the programming community does now. We keep building general-purpose languages that encourage us to ignore problems that don't fit neatly into the chosen paradigm — threading everything through the synchronous, sequential model and claiming that "state is bad," when state is only bad when used ad-hoc in a language that doesn't support reasoning about it.

**Why this matters**

Humanity has been here before. Ptolemaic cosmology was overturned by Galileo. Fixed-capability ICs gave way to the CPU — a narrow-waist sequencer that launched Computer Science, but also locked in assumptions we've carried for decades. Nobel laureate Ilya Prigogine argued that functional thinking set physics back by 100 years. Richard Feynman escaped mathematical notation and invented Feynman Diagrams. "Dark matter" is a tacit admission that our notations are not making cosmology easier.

The incorrect belief driving modern programming: mathematics is the universal language, so it is a natural fit for reprogrammable machines. Programming languages are subsets of a programming toolchain. We need to compose solutions using many languages and tools, not one general-purpose language at a time.
