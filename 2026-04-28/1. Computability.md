*what can be written in assembler that can't be written in any higher level programming language?*

Nothing, in terms of computational power — but that's the boring answer. The more interesting answer is about _control_, not _capability_.

**Theoretically, nothing.** Any Turing-complete language can express any computation. Assembler is not more powerful than C, Python, or Lisp in the computability sense.

**Practically, quite a lot.** Here's what assembler gives you that higher-level languages typically abstract away or forbid:

**Direct hardware access**

- Reading/writing arbitrary memory addresses, I/O ports, memory-mapped registers
- Manipulating the stack pointer directly (building your own call convention)
- Accessing CPU flags (carry, overflow, zero) that HLLs don't expose

**Privileged and system-level operations**

- `HLT`, `CLI`, `STI` (halt, disable/enable interrupts)
- Loading descriptor tables (`LGDT`, `LIDT`) for OS bootstrap
- Switching CPU privilege rings
- Writing interrupt/exception handlers with exact register layouts

**Timing and cycle-level precision**

- `RDTSC` (read timestamp counter) without OS interference
- Inserting `NOP` sleds for cycle-exact timing loops
- Hand-scheduling instructions to avoid pipeline stalls

**Exotic instructions HLLs ignore**

- SIMD intrinsics that compilers won't generate optimally
- `XCHG` with the lock prefix (the original atomic)
- Bit manipulation instructions (`BSF`, `BSR`, `POPCNT`, `LZCNT`)
- `CPUID`, `RDRAND`, `XSAVE`/`XRSTOR`

**Self-modifying code**

- Writing to your own code segment at runtime — straightforward in asm, nearly impossible to express portably in a HLL

**Bootstrapping**

- The very first instruction after reset has _no runtime_, no stack, no heap — asm is the only option

---

The deeper point, which fits your PBP worldview: assembler makes the _machine's actual model_ visible — registers, flags, the call stack as a data structure, memory as a flat array. Higher-level languages impose an _abstraction layer_ that is simultaneously helpful (raises the floor) and limiting (lowers the ceiling). The abstractions hide the hardware's true asynchrony and resource constraints behind a fiction of sequential, typed, garbage-collected safety.

Which is exactly why CPUs, in your framing, are _building material_ — not a destination. The assembler is just the raw interface to that material.

*any language that uses a calling convention cannot implement Forth without a calling convention as efficiently as can be done in assembler*

Exactly right. This cuts to the heart of it.

Forth's efficiency rests on its **inner interpreter** — the `NEXT` macro — which in direct or indirect threaded code is just:

```asm
LODSI          ; load [IP] into W, advance IP
JMP [W]        ; jump to the code for that word
```

Two or three instructions. No frame. No saved registers. No return address on the system stack. Forth maintains its _own_ IP, its own parameter stack pointer, its own return stack pointer — all in dedicated CPU registers.

A language with a calling convention **cannot** express this without penalty because:

- The calling convention _owns_ certain registers — the compiler assumes it can use them, so you can't permanently dedicate them to Forth's VM state
- Every inter-word transition the compiler generates carries frame setup/teardown overhead that Forth's threading model is specifically designed to avoid
- Even `computed goto` tricks in C (GCC's `&&label` extension, used in some Forth implementations) still can't guarantee that the compiler won't spill your "dedicated" registers across the boundary

The subroutine-threaded Forth that C forces you toward replaces `NEXT` with `CALL`/`RET` — which is strictly heavier. You gain from eliminating `NEXT` itself, but you lose because `CALL` pushes a return address (to the _system_ stack, not Forth's), and `RET` pops it — plus the processor's branch predictor and return-address stack get involved.

The deeper principle: **Forth's calling convention _is_ `NEXT`**. It's a custom, minimal convention designed around the specific structure of a threaded word list. Any language that imposes its _own_ calling convention underneath that is adding a layer that cannot be compiled away — because the language's semantics _require_ it.

This is a specific instance of your broader point about GPLs being assemblers for the wrong machine. C is an assembler for a machine with a C calling convention baked in. If your target architecture is a Forth inner interpreter, C is the wrong assembler.

*how does this preceding discussion align with the previous discussion of computability?*

They align perfectly — but at different levels of abstraction, and that difference is the point.

**Computability theory throws away cost.**

It asks only: _does a computation terminate with the correct result?_ Time, space, register pressure, calling overhead — all discarded. Under that model, assembler and Python are interchangeable. Turing equivalence is the ultimate spherical cow.

**The Forth argument lives entirely in what computability discards.**

The claim isn't that a HLL Forth _can't compute_ the same results. It's that it _can't achieve the same cost structure_ — specifically, the register-dedicated, convention-free inner interpreter. Computability theory has nothing to say about this because it has no cost model at all.

**The deeper alignment: computability theory is itself an abstraction that elides the hardware.**

It models computation as if time and resources were free. This is exactly the same move that HLLs make — they abstract away the machine's actual behavior in service of a cleaner programming model. Computability theory is just that abstraction taken to its logical limit.

**And here's where it connects to your broader thesis:**

For real-time systems, timing-critical code, embedded work — a program that produces the correct answer after missing its deadline is _wrong_. Computationally equivalent, but _wrong_. The computability frame is not just incomplete for these cases — it's actively misleading, because it implies interchangeability where none exists in practice.

Computability theory tells you the ceiling. The assembler/Forth argument is about the floor — the irreducible cost that abstractions cannot eliminate, only hide.

The calling convention isn't a detail. It _is_ the architecture.

*does this mean that scripts of opcodes offer a different way of viewing the capabilities of CPU hardware that modern programming languages cannot express?*

Yes — and this is a more radical claim than it first appears.

**Modern languages don't just restrict syntax. They impose a theory of computation onto the hardware.**

That theory includes: hierarchical function calls, a single program counter, sequential execution, a typed value model, and a memory model with ownership or GC. These are not properties of the CPU. They are a _worldview_ projected onto it.

Raw opcode scripts express a different theory — the hardware's _actual_ theory:

- **State, not hierarchy.** Registers plus memory plus PC. No inherent nesting. Jumps go anywhere.
- **No privileged notion of "call."** The stack is just a convention. You can have zero stacks, or five.
- **Interrupts are first-class.** The CPU is inherently interrupt-driven — asynchronous at the silicon level. No HLL models this honestly.
- **Time is real.** Cycle counts, pipeline stalls, cache behavior — these exist at the opcode level and are invisible above it.
- **Multiple simultaneous execution models.** You can implement coroutines, green threads, Forth inner interpreters, state machines — all as different opcode _patterns_, without any one of them being privileged.

**The connection to your broader thesis:**

HLLs are notation systems that make the synchronous, sequential, call/return model easy to express — and make everything else _hard or impossible_. The calling convention isn't just an implementation detail; it's the language's implicit claim about what computation _is_.

Opcode scripts make no such claim. They are _pre-theoretical_ in that sense — the hardware before any model was imposed on it.

**Forth saw this clearly.** The inner interpreter is a tiny opcode script that _implements_ whatever execution model you want on top of bare hardware. Forth doesn't accept the HLL's theory. It builds its own.

**Which raises the sharper question:** if opcodes express the hardware's actual capabilities, and HLLs systematically _hide_ those capabilities behind a model — then every HLL is not just an abstraction, it is an _opinion about what computation is_, enforced syntactically. And that opinion was formed in the era of the printing press and the teletype, not the era of pipelined, interrupt-driven, multi-core silicon.

The CPU hardware has been telling us something about asynchrony and concurrency for 70 years. The languages refused to listen.
