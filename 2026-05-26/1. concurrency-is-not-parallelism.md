# Concurrency Is Not Parallelism

In the English language, the words "concurrency" and "parallelism" mean essentially the same thing.

This is not the case when these words are used when used in computer science dealing with physical hardware.

Let's examine what multi-tasking looks like when mapped onto physical hardware.

# Analogy

Let's begin with an analogy - what does a data _record_ look like in memory:

![[concurrency-is-not-parallelism-struct analogy.drawio.svg]]

A "struct" (class, record, etc.) written in code looks benign. 

When mapped onto 64-bit hardware, though, and emitted as "aligned" data, there are gaps in the memory layout of the fields of the struct.

Some languages / compilers allow programmers to "pack" the data to eliminate the memory gaps. This saves memory, but costs more in compute cycles, because the CPU needs to work harder to access data that is not aligned on 64-bit boundaries.

# Multi-Threading

A similar kind of thing happens with multiple threads, except the packing happens in the dimension of "time", and optimizes for CPU downtime.

Each "thread" is just an isolated "function". Such functions can call other functions contained in the same thread in a synchronous, sequential manner.

We need to add extra software to divvy up CPU time and give slices of CPU time to all threads running on the same computer.

![[concurrency-is-not-parallelism-multi-threading.drawio.svg]]

This happens so quickly that humans perceive the various threads as running in parallel, but, in reality, they run sequentially and slower than they could, due to the time spent running the overhead code (the operating system).

We've been squeezing blood out of this model by adding complex caches, multiple cores, cache coherency, etc. In the end, it's all the same, one computer does not run multiple threads in a truly parallel manner. 

Code running in L1 caches of separate cores can run in parallel, but, become sequential and synchronous as soon as they access data outside of the L1 caches.

# Parallel

![[concurrency-is-not-parallelism-parallel.drawio.svg]]

To create truly parallel threads, you need to devote one CPU for each thread.

In this case, you don't even need the overhead of using an operating system on each machine.

This was the goal of hardware architects in the early days of computing, but CPUs (and whole computers) were too  expensive to justify building computers this way.

Today, though, CPUs are essentially free. Arduinos, ESP32s, Raspberry PIs, etc. are dirt cheap. We can stop using complicated software solutions that save CPU cycles on cheap machines. We can stop wasting memory using old-fashioned techniques.

The internet works this way. Internet nodes are often separate, totally isolated computers that work at their own speed and send blobs of data at each others as messages.

Yet, we continue to use old-fashioned ideas for programming because, as Grace Hopper says "we've always done it this way" (https://programmingsimplicity.substack.com/p/capt-grace-hopper-on-future-possibilities?r=1egdky)

# PBP - Parts Based Programming
PBP (0D) is an attempt to simulate software meant to be run in parallel. 

PBP still fakes parallelism on a single machine (as does the current notion of multi-threading), but results in software designs that are ready to be split off onto separate machines.

The key notion is that synchronous functions cannot (as in "impossible") express asynchronous activities.  Fake parallelism, expressed as concurrency in the functional paradigm, only works the way that Disney cartoons work - by fooling human perception into perceiving motion / parallelism that isn't really there. In the early days, when all that we had were single, stand-alone computers, we made a choice to use small-grained memory sharing instead of message passing. We got away with that choice for a long time. Yet, the choice has been causing us great deals of accidental complexity, starting with the notion of "thread safety". The issues of thread safety simply evaporate when isolated computers are used. Programming complexity drops drastically when thread safety no longer needs to inflict itself on our thought patterns, and on our programming language designs and on our perceived need for operating systems. 

For example, early, cartridge based gaming systems didn't used operating systems and were fundamentally simpler to program. In essence, the invention of "operating systems" was just a soft version of plugging and unplugging cartridges. We wished that we could have systems with multiple slots, each with its own, isolated CPU, with maybe big, fat, mechanical rotary switches to switch shared hardware resources (like CRT displays, keyboards, joysticks) between programs.
