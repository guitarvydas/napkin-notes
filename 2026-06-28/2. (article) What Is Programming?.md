# What Is Programming?

**TL;DR** Programming is not defined by functions, calling conventions, or any other abstraction invented for one narrow problem domain. Programming is the act of specifying executable behavior for an agent. Computation is one kind of behavior. Sequencing, interaction, control, and music generation are others. Mainstream languages address only the computational slice, and their conceptual grip on the field distorts how programmers think about every other kind of problem.

---

## L;R

### The dictionary answer

The dictionary defines programming as "creating or specifying an ordered set of instructions or actions that achieve a desired result." That definition pre-dates computers. Originally a program was a schedule of events — a theatre program, a radio program, a concert program. When electronic computers appeared, engineers borrowed the word because they were literally scheduling operations for a machine.

Notice what the definition does not say. It says nothing about functions. It says nothing about calling conventions, activation records, return values, or stack frames. It says nothing about computation at all.

### Programming is not the same thing as computation

I can program a computer to perform a calculation. I can also program a computer to play music, to become a musical instrument, or to sequence a manufacturing process. Only the first of those is naturally described as a computation. The others are ongoing behaviors unfolding in time. A digital piano does not compute "42." It waits for a key press, synthesizes sound, and continues indefinitely.

The classical notion of computation — the evaluation of a mathematical function from inputs to outputs — is a subset of programming, not its definition. Programming is the specification of behavior. Computation is one behavior that can be specified.

### When I write music, am I programming?

Consider what a piano score actually is. It specifies actions. Those actions occur in a particular order. There are control structures — repeats, first and second endings, D.C. al Fine. There are even conditional interpretations in some scores. The processor is a human performer.

We do not call this programming because our language evolved specialized words for different instruction sets. A set of instructions for a computer is a program. A set of instructions for a musician is a score. A set of instructions for a cook is a recipe. A set of instructions for an actor is a script. The underlying abstraction is the same in every case: a symbolic description that guides an agent's behavior.

The distinction between program and score is social, not technical.

From an information-theoretic viewpoint, a Beethoven sonata is strikingly program-like. The notation is source code. The pianist is the interpreter. The piano is the hardware. The performance is the execution. The main difference is that the human interpreter intentionally leaves room for interpretation, whereas most computer programming strives to eliminate ambiguity.

### The word "programming" is overloaded

Programming a computer, programming a thermostat, programming a TV schedule, programming a robot, and programming a concert all use the same word. They share the idea of prearranging future behavior, but they differ significantly in what is being arranged and how precisely it must be specified.

A richer vocabulary would help. We might distinguish behavioral specification (defining desired behavior), machine programming (specifying behavior for an automatic machine), performance specification (specifying behavior for a human performer), scheduling (specifying temporal order), and configuration (specifying parameter values). These are related activities with a common abstract structure, not a single activity with a single set of appropriate abstractions.

### Mainstream languages address only a slice

If we classify programming by intent, we get something like this:

- Computational: "What value should be produced?"
- Sequential: "What should happen next?"
- Reactive: "How should the system respond?"
- Control: "How should this process be regulated?"
- Interactive: "How should this conversation evolve?"
- Generative: "What artifact should be produced?"

Programming a calculator belongs to the first row. Programming a piano belongs to the second and third. Programming a machine sequencer belongs to the first and fourth. Mainstream general-purpose languages — C, C++, Java, Python, JavaScript, Rust, Go, and the rest — are organized around the first row. Their primary decomposition mechanism is the function, and everything else is layered on top of that.

This is not a coincidence. These languages inherited their conceptual foundations from languages developed for scientific computing in the 1950s and 1960s. Those languages were designed to express numerical algorithms. As computers spread into real-time control, music, user interfaces, networking, and embedded systems, the same programming concepts were extended rather than fundamentally reconsidered.

The result is that programmers routinely express behavioral systems — systems where the natural structure is temporal, event-driven, or concurrent — through computational abstractions that do not match the problem. Event loops, callbacks, state machines, coroutines, async/await, and reactive frameworks are all attempts to express behavior over time using languages whose core abstractions were originally optimized for the computation of values. They work, but they introduce accidental complexity. The translation from the conceptual model to the programming model obscures the structure of the problem.

### CALL and RET do not implement functions

A CPU does not implement functions. It implements mechanisms from which function calls can be constructed, along with many other execution models.

A CALL instruction says: save where execution should resume, then continue execution somewhere else. A RET instruction says: resume execution from the saved location. That is all. Nothing in that constitutes a mathematical function.

A language runtime then layers additional conventions on top: parameter passing, stack frames, local variables, register preservation, return values, recursion. These conventions have become so ubiquitous that programmers mistake them for properties of computation itself. They are not. They are one solution to one problem — how independently written pieces of code can cooperate — and they are not the only solution.

Subroutines were originally invented to save memory by creating shareable code. A subroutine could print a character, start a motor, update a display, wait for an interrupt, or never return. Many early assembly-language subroutines were not mathematical functions. They were reusable control transfers. The function abstraction was built on top of that mechanism through calling conventions. The hardware was never function-oriented; it was made function-oriented by agreement.

Forth demonstrates the alternative. Words communicate through an explicit data stack. Composition is based on stack effects, not on activation records and named parameter lists. The calling convention is extremely lightweight. Prolog demonstrates another alternative. Its basic abstraction is not function application at all. It is relations, unification, search, and backtracking. Execution is driven by proof search. Trying to describe Prolog in terms of functions misses its essence. Hardware description languages describe concurrent signals, propagation, and timing. PLC ladder logic describes contacts, coils, scans, latches, and timers. None of these are organized around function calls.

### There is no required fundamental unit

Programming language research has long searched for the right abstraction — procedures, objects, functions, actors, processes, logic, dataflow, effects, capabilities. The underlying assumption is that there exists some sufficiently expressive core.

That assumption is worth questioning. Mathematics does not have one primitive abstraction. Depending on the problem, mathematicians think in terms of functions, sets, relations, groups, graphs, categories, differential equations, or probability spaces. No mathematician insists that graph theory must be reduced to functions before it qualifies as real mathematics. We choose the notation that best matches the structure of the problem.

Engineering does not have one universal design unit either. An electrical engineer thinks in signals and transfer functions. A mechanical engineer thinks in forces and constraints. An architect thinks in spaces and circulation. Each discipline has abstractions that fit its domain. Programming may simply be another engineering discipline with multiple subdomains, each deserving its own natural notation.

### The real mistake is premature unification

Modern languages broaden themselves by adding features — async/await, pattern matching, streams, coroutines, actors, reactive extensions, effect systems. But the core decomposition does not change. A program is still broken into functions, which are implemented by calls. Every extension is layered on top of that structure.

This is broadening by addition rather than by reconsidering the model. The center of gravity remains computation even when the problem is behavior.

The mistake is not that functions exist. Functions are excellent for the domains where functions fit. The mistake is treating functions as the universal unit of programming. That is like insisting that every kind of writing be expressed as prose. Prose works well for novels, essays, and journalism. It is awkward for music, architecture, chemistry, dance, and mathematics. Those domains invented their own symbolic languages because prose is not the best notation for everything.

Perhaps programming has made the opposite mistake: assuming that one symbolic language should describe every kind of executable behavior.

### Composing multiple notations

The productive direction is not to invent yet another language that handles everything through a different universal abstraction. It is to build systems that compose multiple notations, each optimized for a different class of executable behavior.

UNIX already demonstrated this philosophy. A UNIX pipeline does not care whether one process is written in C, another in awk, another in Python, and another in shell. The composition mechanism lives outside the individual languages. The profound idea is that the unit of construction is not a program in a language but a component with defined interfaces that can be wired to other components regardless of their internal implementation.

The limitation of UNIX at the process level is overhead — process creation, context switching, operating-system mediation. The goal would be to achieve the same heterogeneous composition at much lower cost, with semantic boundaries rather than operating-system boundaries.

An orchestra already does this. The conductor reads a full score. The violinist reads a violin part. The percussionist reads percussion notation. Each is a specialized language. The miracle is not that they all use the same notation — they do not. The miracle is that they synchronize. Synchronization is the composition mechanism.

The research question worth asking is not "what is the one right abstraction for programming?" It is "what composition mechanism allows each part of a system to be expressed in the notation that best fits it, while permitting the parts to cooperate efficiently?"

That question shifts the emphasis from inventing a better universal language to inventing a better composition mechanism. It treats programming as a family of activities whose common feature is the specification of executable behavior, not the common use of one particular notation inherited from numerical computation in the 1950s.

Programming is not defined by functions. Functions are one tool for one domain. The field is larger than the tools we happen to have standardized on.
