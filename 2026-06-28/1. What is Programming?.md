The word comes from **program**, which originally meant:

> A written plan or schedule of events.

That, in turn, derives from the Greek **programma**, meaning:

> "A public notice," or literally "something written beforehand."

The suffix **-ing** turns the noun/verb into the activity:

> **programming** = creating or specifying a program.

> [pt] was the definition different in the 1960s, 1970s, 1980s, i.e. before "functional programming" became viable?

---

Yes, but perhaps not in the way you might expect. The meaning of **"programming"** itself changed very little from the 1960s through the 1980s. What changed was **what people considered a "program" to look like**.

Early dictionaries typically gave definitions like:

> **Programming**: The process of preparing a program for a computer.

or

> **Programming**: The planning, coding, testing, and debugging of a computer program.

These definitions are broad enough to include any style of programming.

### What did "program" imply at the time?

The implicit mental model was often:

> A sequence of instructions executed by a machine.

This fit the dominant languages and hardware of the era:

- Assembly
    
- FORTRAN
    
- COBOL
    
- ALGOL
    
- BASIC
    
- C
    

Programs were generally viewed as sequences of statements that modified machine state.

### Functional programming already existed

Interestingly, **functional programming did not become viable only after the 1980s**. Its theoretical foundations are much older:

- Lambda calculus was introduced by Alonzo Church in the 1930s.
    
- LISP (1958) was a practical functional language from the beginning, although it also supported mutation.
    
- ML appeared in the 1970s.
    
- Purely functional languages became more practical during the 1980s and 1990s with languages like Haskell.
    

So functional programming predates much of what people think of as "modern programming"; it simply wasn't the dominant industrial style.

### How definitions evolved

The main shift occurred in textbooks and programming-language research rather than dictionaries.

A 1970s programming text might describe programming as:

> Writing a sequence of instructions to control a computer.

A modern programming-language text is more likely to describe programming as:

> Specifying a computation.

That subtle change is important:

- **Sequence of instructions** emphasizes _how_ the computation proceeds (imperative programming).
    
- **Specification of a computation** emphasizes _what_ computation is performed (declarative, functional, logic, dataflow, constraint, and other paradigms).
    

### In hindsight

Today, computer scientists generally view a **program** as a mathematical or formal description of a computation. Execution models—imperative, functional, logic, concurrent, dataflow, probabilistic, and others—are different ways of giving meaning to that description.

So while dictionaries didn't substantially change their wording, the conceptual understanding within computer science broadened from **"a list of instructions"** to **"a formal specification of a computation."** That broader view comfortably encompasses imperative, functional, logic, and many other programming paradigms.

That's a deep question, and philosophers of programming languages have asked essentially the same thing.

The short answer is: **it could be called programming**, but historically we reserved the word for programming _machines_.

Consider your definition:

> Creating a set of instructions or actions that achieve a desired result.

A piano score certainly qualifies.

- It specifies actions.
    
- They occur in a particular order.
    
- There are control structures (repeats, first/second endings, _D.C. al Fine_, _Dal Segno_, codas).
    
- There are even conditional interpretations ("play ad lib.", "repeat if desired") in some scores.
    
- The "processor" is a human pianist.
    

From an information-processing perspective, a musical score is very much like a program.

### Why don't we call it programming?

Because the word **program** had a different history.

Originally, a **program** meant a schedule or plan:

- a theatre program
    
- a radio program
    
- a concert program
    

When electronic computers appeared, engineers borrowed the word because they were literally creating a schedule of operations for the machine.

Thus "programming" became associated with machines rather than with any arbitrary set of instructions.

### The distinction became social rather than technical

Suppose I write:

```
Turn left.
Walk 20 m.
Open the red door.
Wait 5 seconds.
```

That's a program—for a person.

Or:

```
1. Crack two eggs.
2. Beat thoroughly.
3. Bake at 180°C for 25 minutes.
```

That's a program—for a cook.

We don't usually call recipes or musical scores "programs" because our language evolved specialized words for different instruction sets:

|Instructions for...|We call it...|
|---|---|
|Computer|Program|
|Musician|Score|
|Cook|Recipe|
|Actor|Script|
|Builder|Blueprint or plans|
|Orchestra|Score and parts|
|Robot|Program|

The underlying abstraction is the same: **a symbolic description that guides an agent's behavior**.

### Computer science has broadened the concept

Modern theoretical computer science often defines a program much more abstractly:

> A formal description of a computation or behavior.

Notice it doesn't say "computer."

In fact, fields such as:

- denotational semantics,
    
- operational semantics,
    
- process calculi,
    
- rewriting systems,
    

all treat a program as a formal object independent of whether the executor is silicon, a person, or even another mathematical system.

### An interesting historical irony

The earliest programmers didn't always think of themselves as "writing code." They often said they were **preparing instructions** or **coding**. Likewise, composers have long been preparing symbolic instructions for performers.

From an information-theoretic viewpoint, a Beethoven sonata is remarkably program-like:

- the notation is the source code,
    
- the pianist is the interpreter,
    
- the piano is the hardware,
    
- the performance is the execution.
    

The main difference is that the "interpreter" (the human performer) intentionally leaves room for interpretation, whereas most computer programming strives to eliminate ambiguity.

So I'd argue that **"programming" has two meanings**:

1. **The historical, everyday meaning:** writing instructions for an automatic machine (usually a computer).
    
2. **The more abstract meaning:** creating a symbolic specification of behavior for an executor.
    

Under the second meaning, writing music, choreographing a dance, devising a recipe, and writing software are all instances of programming—they differ primarily in the nature of the executor and the degree of precision required.

> [pt] Is music a computation?

That depends on what you mean by **computation**. There are two major perspectives.

### Classical view: No

Traditionally, a computation is the evaluation of a mathematical function or the execution of an algorithm.

For example:

- sorting a list
    
- multiplying matrices
    
- solving a system of equations
    
- rendering an image
    

Playing a piano piece is not usually considered a computation because its purpose is not to compute a mathematical result. It is a performance.

### Modern computer science view: Yes

Many researchers now define computation more broadly as **the evolution of a system according to rules**.

Under this view:

- A CPU executes machine instructions.
    
- An interpreter executes a program.
    
- A musician executes a score.
    

The score specifies a sequence of symbolic actions, and the performer transforms that symbolic representation into sound over time. That is a form of information processing.

### Is the pianist "computing"?

A pianist performs many computations internally:

- Decoding musical notation.
    
- Mapping notes to finger movements.
    
- Timing events.
    
- Coordinating both hands.
    
- Interpreting tempo and dynamics.
    
- Producing acoustic output.
    

These are genuine computations occurring in the nervous system.

However, **the musical performance itself is not usually regarded as "the computation."** Rather, the performance is the observable behavior resulting from many computations.

### An analogy

Suppose you have these three systems:

```
Computer
Source code
    ↓
CPU
    ↓
Pixels on a screen
```

```
Player piano
Piano roll
    ↓
Mechanism
    ↓
Music
```

```
Human pianist
Musical score
    ↓
Brain + body
    ↓
Music
```

All three involve a symbolic description being interpreted to produce behavior. The difference is mainly in the nature of the interpreter.

### Where the analogy breaks down

Computer programs are generally designed to have a single, well-defined meaning. If two compliant implementations execute the same program, we expect essentially the same behavior (allowing for timing and platform differences).

Musical scores deliberately leave room for interpretation. They often omit or underspecify details such as:

- exact tempo (unless a metronome marking is given),
    
- timing nuances (rubato),
    
- voicing,
    
- phrasing,
    
- pedaling,
    
- tone color.
    

Two world-class pianists can play the same score very differently while both being faithful to it.

In programming-language terms, a musical score has **intentional nondeterminism**: multiple executions are considered correct.

### A deeper viewpoint

Some philosophers of computation argue that **computation is the execution of symbolic rules**, regardless of the physical substrate. From that perspective, performing music is one instance of computation. Others reserve the term for processes that manipulate symbols to compute formal functions, in which case musical performance is not itself a computation, though it relies heavily on computation by the performer.

So the answer is not settled by definition. If you adopt the broad notion of computation as **rule-governed symbolic execution**, then a musical performance can reasonably be viewed as a computation. If you adopt the narrower mathematical notion of computation as **algorithmic evaluation of a function**, then it is not. The broader definition has become increasingly influential in areas such as programming language theory, distributed systems, and models of interactive computation.

## The functional viewpoint

Pure functional programming models programs as mathematical functions:

```
f : Input → Output
```

This is an elegant model for many problems:

- symbolic computation
    
- compilers
    
- parsers
    
- optimization
    
- numerical algorithms
    

But it becomes strained when the essence of the program is not producing a value but maintaining a relationship with the outside world.

Music is a good example.

A synthesizer is not trying to compute

```
Sound = f(Keys)
```

Instead it is engaged in an ongoing dialogue:

```
Key down
↓
Start oscillator

Key released
↓
Begin envelope release

Pedal pressed
↓
Hold notes

...
```

This is a conversation, not a calculation.

## The workarounds

Functional languages certainly _can_ express these systems, but they often do so by embedding the interaction inside abstractions such as:

- monads
    
- arrows
    
- effect systems
    
- functional reactive programming
    
- event streams
    

These are ingenious, but they are evidence that the underlying model is not interaction itself.

Likewise, imperative languages often struggle to express dataflow naturally, and object-oriented languages can struggle to express concurrent protocols naturally. Every paradigm has domains where it is more or less natural.

## I would sharpen one point

> [pt] CALL and RET were invented to save memory by creating shareable subroutines. Parameterization and Value Returning, required to implement functions, are not built into the hardware. I think that's an important refinement, and it makes your argument more precise.

You're distinguishing between **control transfer** and **function invocation**.

Those are often treated as synonymous, but they aren't.

A `CALL` instruction fundamentally says something like:

> Save where execution should resume, then continue execution somewhere else.

A `RET` says:

> Resume execution from the saved location.

That's all.

Everything else that we associate with a "function" is layered on top.

For example:

- where arguments live (registers? stack? globals?),
    
- how many arguments there are,
    
- whether arguments have names,
    
- whether there is a return value,
    
- where the return value is placed,
    
- whether the routine is recursive,
    
- which registers are preserved,
    
- whether local variables exist,
    
- whether stack frames exist.
    

None of that is implied by `CALL` or `RET`.

In other words,

```
CALL/RET
    ⊂
Function call
```

not

```
CALL/RET
    =
Function call
```

The function abstraction requires an **Application Binary Interface (ABI)** or equivalent convention.
