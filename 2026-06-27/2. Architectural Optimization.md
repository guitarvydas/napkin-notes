Hmm, maybe this leads to the notion of architectural optimization? Presently compilers optimize code. Instead, though, optimizing at a higher level might produce greater gains in efficiency. Thinking... an architect draws and easy-to-read architecture and gets it running, but, on some hardware, it would be more efficient to modify the architecture and make it more messy - like the sync wires running all over the circuit in Pong. Current compilers can't do that, because of the strong coupling implemented by CALL/RETURN. Changing a part of the architecture would change synchronization relationships and produce gotchas. Would it become easier to write high-level, architectural optimizers if the units were more loosely coupled? My knee-jerk reaction would be to write such optimizers in a relational language (e.g. something like Prolog). Actually, it wouldn't even need to be automated. One could put the original architecture in a git branch, then manually modify the architecture, for a specific target machine, in another branch. The original still exists and the provenance from original to the highly tuned one would be captured by git. Hmm...
