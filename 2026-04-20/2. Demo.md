**Your question:** _What is needed for a reasonable demo of PBP/0D? Is "hello world" enough? Given this thread of discussion, what needs to be shown?_

---

"Hello world" is not enough. It demonstrates that a system can produce output, which is the one thing nobody doubts. It shows nothing about the properties that make PBP/0D meaningfully different from what already exists.

Given this thread, the demo needs to make the following visible:

**The things that need to be shown**

The most important is **fan-out and fan-in with multiple ports** — a component that receives a message and sends output to more than one destination, or that receives from more than one source, handled naturally rather than as a special case bolted onto a single-output model. This directly refutes the "one-in-one-out" restriction of the functional paradigm.

Second is **async composition without tight coupling** — two or more components communicating through messages where neither blocks waiting for the other. The point is not concurrency for its own sake but demonstrating that the components are genuinely topology-blind: each component does not know what it is connected to.

Third is **probe insertion** — showing that you can add an observational component to the diagram and watch intermediate state, then remove it, without modifying any of the existing components. This is the debugging argument made concrete. It directly demonstrates that the diagrammatic notation preserves design intent in a way that sync/seq debuggers cannot match.

Fourth is **the Container/Leaf distinction** — a diagram that contains at least one nested sub-diagram, showing recursive composition. This is what schematics lack and what Lisp-like structure provides. Without this, the demo shows a flat circuit, not a scalable architecture.

Fifth, even briefly, is **a leaf node doing something sync/seq** — showing that a TPL component (a normal function, a piece of Python or JavaScript) sits naturally at the leaf of an async composition without modification. This validates the synthesis claim: sync is a participant in async, not its replacement.

**What the demo should not do**

It should not spend time on anything that existing languages already handle well. No string manipulation, no arithmetic, no file I/O in isolation. The moment the demo looks like a program that could have been written more easily in Python, the argument is lost.

**The ideal shape**

A small network of three to five components where: the topology is visible in the diagram, one component fans out to two others, a probe component can be inserted and removed live, and one sub-diagram is expanded to show nesting. The problem being solved matters less than the structural properties being demonstrated. Something like a simple pipeline that splits, processes in parallel paths, and recombines — a shape that is famously awkward in synchronous/sequential code — would make the contrast vivid without requiring domain knowledge to appreciate.

The goal is that someone watching should finish thinking: _this would have been callback hell in JavaScript, and here it was just topology_.
