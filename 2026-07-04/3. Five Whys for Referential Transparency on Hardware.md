# Five Whys for Referential Transparency on Hardware
2026-07-04

Q: Is "referential transparency" a way to compose functions, akin to using LEGO® blocks?
A: Composing functions this way works well on paper, where the concept of time doesn't interfere. 
Q: How does the concept of time interfere with referential transparency applied to software functions on hardware?
A: When this is mapped onto hardware, functions become tightly coupled.
Q: How do functions become tightly coupled?
A: Functions in software appear to pass data from caller to callee, but, in fact they pass data *and* control flow - the caller suspends (blocks) waiting for an answer from the callee. This is control flow. 
Q: Why isn't this an issue on paper but becomes an issue on hardware?
A: The implementation of functions in hardware uses CALL / RETURN which needs a small unit of time (one or more CPU cycles). The unit of time is ignored when the notation is used on paper, but is concretely manifested when implemented in hardware.
Q: Why does this small unit of time matter?
A: Functions are silently synchronized in a manner akin to micro-management.
