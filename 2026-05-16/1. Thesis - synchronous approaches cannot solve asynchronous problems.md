Thesis:
- what's called "programming" today isn't representative of how a CPU IC works
- this mismatch in our notation has led to bloat - bloated footprint (Mb of RAM instead of Kb of RAM) and bloat in in the way that we think about problems
- this way of programming was chosen from a number of options in the early days of programming by the those who believed that mathematical notation - meant to be use on paper - was the best way to deal with creating machine code for CPU ICs
- we didn't notice the mismatch over the past few decades because the problems and solutions in the early days were aimed at single-threaded, sequential, synchronous problem domains and because economics made it cheaper to expend human brain power instead of expending hardware
- standing on the shoulders of these early "giants" is suitable for solving problem in domains that are sequential and synchronous, but are not suited for solving problems in domains that are asynchronous and non-sequential
- today, we have more choice, the economic situation has flipped - CPUs are cheaper than human development time, yet, we continue to use outdated methods invented for different problem domains and we call it "progress" to waste time force-fitting solutions into the old fashioned methods. This force-fitting causes us to invent complicated work-arounds and to believe that they are required. 
- For example, a lot of complication comes from the basic principles of "thread safety". What we call concurrency is based on techniques, called "time-sharing", invented to squeeze every ounce of CPU processor time out of each CPU, whereas it would be immensely less complicated to use multiple CPUs and not to have to worry about "thread safety" at all. This is what was originally imagined for the use of CPUs by hardware people, but, was impossible due to the high cost of CPUs in the early days
- This old-fashioned, paper-only, methodology leads to closed system thinking, which is not suitable for addressing problems which are asynchronous and non-sequential, like the internet. One example of the gotchas that arise out of applying old-fashioned methodology to current asynchronous and non-sequential problem domains was the rise of "callback hell" - a surprise to practitioners. The gotchas were papered-over using band-aids and more complication - like promises, `.then`, etc. But, these are just band-aids that allow us to keep using an usuitable notation for these kinds of problems - they basically deny that there might be other ways to attack these problems using different models, notations and methodology
- can't solve internet with shared memory, cache coherency approaches meant for sync/seq problem domains, because internet is the opposite of sync/seq ; if we continue to try, we will end up creating accidental complexities, we will waste time, and we can never (as in "never", "impossible") actually solve asynchronous problems, because - by definition - "synchronous" is the exact opposite of "asynchronous". Wrapping our asynchronous problems in synchronous straight-jackets causes us to solve only problems that can be thought about in a synchronous manner. This is like astrophysics and cosmology assuming that the universe is well-understood, but missing the point that 95% of the problem domain is brushed aside using catch-all phrases like "dark matter".
- if we truly want to solve a plethora of more modern problems in asynchronous and non-sequential domains, we have to stop using old-fashioned, complicated, tower-of-band-aid techniques and look for less-complicated ways to deal with modern problem domains that are more suitable to the domains themselves
- we have to stop using the idea that we must hammer every solution into a generalized notation and we need to allow ourselves to use multiple notations / programming languages in our workflows
- our workflow needs to be Turing complete, but each language within the workflow does not _need_ to be Turing complete. If we stop tinkering with our old-fashioned notation, we might be able to invent workflows that support multiple languages. After all, programming languages are simply cave-man IDEs and workflows invented in the early days. A  programming language is considered "good" if it can support multiple paradigms, but, that is just a lingering after-effect of our attempts to develop multi-paradigm workflows under the misconception that our full workflow needs to be squeezed into a single programming language
- we continue to invent new programming languages that abstract programming issues, but all of those languages are based on the underlying paradigm of synchronous / sequential operation, hence, such "new" programming languages (C, Haskell, Rust, Clojure, etc.) are all essentially the same, forcing us to want to solve asynchronous problems using an unsuitable base paradigm, causing us to waste time and brain power. Instead we could be using bowls of inexpensive CPUs to solve non-sequential problems.
- Asynchronous problems cannot be cubby-holed into synchronous thinking. You might succeed in weeding out only the synchronizable instances of problems and treating them in a synchronous way, but that leaves 95% of the problem space unsolved. And it leaves open the possibility of encountering more future, ad hoc road-blocks, like callback hell and the Mars Pathfinder fiasco, on your path
- Continuing to strive for "General Purpose" languages is a death march
- in the 20th century, electronics formed a stepping stone towards synchronous, sequential programming
	- now, in the 21st century we need to use what we've developed as a stepping stone towards asynchronous, non-sequential system composition
	- 20th century techniques have given us very cheap CPUs	
	- 20th century techniques have given us very cheap memory
	- 20th century techniques have given us very a view of larger, asynchronous, non-sequential problem domains, but, also, have given us _tells_ that the 20th century techniques aren't good enough to solve these larger problems
	- 20th century electronics showed us that we needed something even more flexible than physical electronic ICs, so we invented soft-hardware and named it "programming", but, we should be able to see that 20th century "programming" isn't uncomplicated enough to step up to solving 21st century problem domains
	- "programming" doesn't use the techniques of hardware, I think that we need to blend the two approaches
