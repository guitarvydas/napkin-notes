- humans invent and build all sorts of devices that help augment human capabilities
- for example, wood is used to build houses, fences, etc.
- electronic components are used to build devices such as $7.95 Radio Shack calculators, keyboards, mice, video displays, computers, etc.
- a CPU is a specific type of electronic component that can have its behaviour changed by reprogramming it with scripts of opcodes (the CPU IC must be bolted to a memory IC)
- CPUs help us invent new hardware devices more quickly
- "software" is the the name we use for scripts of opcodes
- software is just soft hardware, i.e. just a building material
- computer science is the formalization of *one* style of creating opcodes, as well as an extension to mathematics into the digital domain
	- software written in a function-based style, i.e. just about any current programming language like Python, Javascript, Rust, etc. is a way to create scripts of opcodes that simplifies the act of creating computation devices where only the result matters, at the expense of ignoring memory footprint and control flow
	- there are other ways to create scripts of opcodes that simplify the design of other kinds of devices, but, we've been focussing for decades mostly on the use of the function-based style (impure FP)
	- the epitome of the function-based style is Functional Programming
		- pure FP is so far removed from the way that actual electronic hardware can work, as to be considered a domain unto itself
			- we've figured out how to map FP onto electronic machines and CPU ICs
		- the beginnings of FP was Lisp 1.5, improved upon by modern Sector Lisp
		- the cementing of the function-based style into the psyche of a generation(s) of programmers was the invention of the language C and the idea of calling conventions ("calling" implies "function calling", some languages, like Forth and Prolog, don't use "function calling", hence are hindered by the requirement to adhere to "calling conventions")
			- C rose to popularity based on false advertising
			- UNIX is said to have been written in C, while UNIX (V1-V3) was actually first written in assembler
			- C is said to be "close to the hardware", but it is only close to PDP-7 hardware and doesn't map very cleanly onto other styles of CPU ICs.
			- The C (K&R C) used in the early days of UNIX is not even compatible with the modern language that we call C (ANSI C). K&R C was a *tool* for building scripts of opcodes for a specific kind of CPU, whereas ANSI C is an anal-retentive tool for type-checking scripts of opcodes to ensure portability between different kinds of CPUs almost none of which are PDP-7-like
				- even though it hasn't been measured that type-checking actually improves productivity - [Smallshire]([https://www.infoq.com/presentations/dynamic-static-typing/](https://www.infoq.com/presentations/dynamic-static-typing/)) claims that type-checking only catches 3% of real errors
			- before C (and Pascal) there was no concept of "calling convention" - use of  "calling conventions" implies the sole use of "functions" while discouraging the invention of different kinds of uses of opcode scripts.
