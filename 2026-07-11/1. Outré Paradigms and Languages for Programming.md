# function-based approach
- trade off memory for functional notation
- disregard timing (ordering, side-effects) to support functional notation
- needs scaffolding - called "operating systems" - to support functional notation
- cannot express (i.e. impossible) asynchronous parallelism
	- what's called "parallel", "concurrent", "multi-tasking" is actually time-sharing, i.e. fake concurrency
	- continued notation worship has led us down blind alleys resulting in callback hell, Mars Pathfinder fiasco, etc.

# DI vs Code
DI is for humans, code is for machines (and Production Engineers interested in extreme optimization)

"DI" means Design Intent, aka "software architecture" and explaining what the software architect intended for the machine to do, as opposed to reverse-engineering the code and theorizing about what the DI was meant to be. Kind of like disassembling code using a disassembler.
# Big Programs
## current approach
make bigger programs by increasing sprawl, which implies the need for type-checking
## overlooked approach 
make bigger programs by composing small, understandable chunks (no need for type checking, since it remains "obvious")

- early programs, e.g. BASIC, used to fit on one eye-full (a screen, a window)

# Memory Sharing
- requires lots of extra code for handling "thread safety"
- applicable only to only a small part (guesstimate 5%) of problem space, e.g. graphics, fat / rich media where efficient bulk data transfer is required
- not applicable to problems like distributed computing and internet, because messages are thin and don't require optimization for bulk data transfer

# Hidden Dependency
- CALL/RETURN in hardware guarantees that functions depend on one another
- impure message passing (including Smalltalk)
	- passes data AND passes control flow (caller suspends / blocks while waiting for callee to return a result ==  control flow restriction)
- makes it impossible to think about asynchronous systems and impossible to use a LEGO-block-like approach, since there's too much tight coupling between components in the form of hidden control flow dependencies

# Parsing Diagrams
- use Prolog (and miniKanren) triples and relational programming to infer relationships between figures
- example [pbp-sm]() uses two Prolog files, 26 lines and 17 lines for inferencing information not provided by draw.io
- example [dtree](https://github.com/guitarvydas/dtree) uses three Prolog file 8, 99, 75 lines long for inferencing information not provided by draw.io
- use drawing editors, like drawio and Excalidraw, that save diagrams in parseable text form, e.g. XML, graphML. The text form is not very human readable, but is easily parsed by machines.
