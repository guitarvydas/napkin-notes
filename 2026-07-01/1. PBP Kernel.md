# The Parts Based Programming Kernel
2026-07-01

The PBP “kernel” is 2 things - 
1. a template registry, 
2. coroutining.  

When you draw a diagram it becomes a template. When you start up a PBP system it does several things:
1. loads ("registers") all of the templates
	- Container parts are loaded from JSON files generated by the drawio diagrams (converted to JSON by `das2json`)
	- Leaf parts are declared by creating Leaf templates using `mkTemplate (...)`, which are then inserted into the registry by `register_component (...)`
2. Starts with the top-most Container component and recursively instantiates unique runtime parts all the way down.
3. Walks the the top-most Container, which recursively walks all children parts down to the Leaf parts. Each Container processes one input *mevent* at a time, going into a loop which recursively tick each child until the child has no more work to do.  Containers can contain Containers, so the process is recursive. When a Leaf part is hit, it doesn't recur any further.

This is akin to what happens in operating systems, except the details tend to be hidden by complicated glop and generally overlooked. The O/S “loads” a program - reads the Elf (or whatever .obj) file, makes various fixups and twiddles addresses and the MPU, then it runs the code. When the code performs a system trap, i.e. CALLs the operating system, it calls a function that twiddles protection levels, and so on, to the let the O/S code run unrestricted. I think that the O/S code sits in permanently loaded memory space, so that you don’t need to fool with the MPU every time you call the O/S. 

If this sounds complicated, it is - unnecessarily so. It comes from the 20th century desire to use only 1 CPU and to time-share it. With the result that if you have 10 apps and 90 O/S threads running, 1 app gets about 1/100 of the CPU (give or take, modulo complicated priority management, etc. The software architect doesn't really get to control what happens. If another app/thread starts up, it takes CPU cycles away from all of the code that is already running. 

It would have been simpler to just put a zillion CPUs on a motherboard, each running exactly one thread, but, giving 100% CPU to each thread. This would have cut out a lot of complexity from our code (e.g. "thread safety"). But, CPUs used to cost as much as a house, so economics influenced our designs. Today, we would have done things differently. In fact, we *should* ease into doing things differently. That is what PBP is about.
