# The Parts Based Programming Kernel

_July 1, 2026_

---

Every system has a kernel — the irreducible core that makes everything else possible. For operating systems, it's the code that loads programs, manages memory, and controls the CPU. For Parts Based Programming (PBP), the kernel is simpler: two things.

1. A template registry.
2. Coroutining.

That's it. Let me explain what that means and why it matters.

---

## What the Kernel Does

When you draw a diagram in PBP, that diagram _becomes_ a template. Boxes become parts. Connections become message routes. The visual is the program.

When a PBP system starts up, it does three things in sequence:

**First, it loads all the templates.** Container parts — the ones that hold other parts — are loaded from JSON files generated by the drawio diagrams (converted by `das2json`). Leaf parts, the atomic building blocks that do actual work, are declared in code using `mkTemplate(...)` and inserted into the registry with `register_component(...)`.

**Second, it instantiates.** Starting from the top-most Container, the runtime recursively creates unique instances of every part, all the way down through the hierarchy to the leaves.

**Third, it runs.** The top-most Container walks its children. Each Container processes one input message-event (mevent) at a time, ticking each child in a loop until that child has no more work to do. Containers can hold Containers, so this walk is recursive. When you hit a Leaf, the recursion stops — Leaves don't contain anything; they just act.

This is a clean, explicit, traceable execution model. No surprises.

---

## Sound Familiar?

It should — because this is almost exactly what an operating system does.

An OS loads a program: reads the ELF file (or whatever object format), fixes up addresses, tweaks the MPU, then hands control to the code. When that code needs OS services, it issues a system trap — a CALL that bumps the protection level and transfers control to OS code sitting in permanently-mapped memory. The OS does its thing, drops back to user-land, and returns.

The mechanics are the same as PBP: a registry (the filesystem + loader), instantiation (process creation), and recursive delegation (the call stack + scheduler).

The difference is in what's been piled on top.

---

## The 20th Century Tax

Operating systems as we know them exist to solve a specific 1960s problem: CPUs were expensive. One CPU, many users. You had to time-share.

Time-sharing is where the complexity explosion happened. If you have 10 apps and 90 OS threads running, each app gets roughly 1/100th of a CPU — give or take, modulo complicated priority management, scheduling queues, and preemption logic. And here's the thing: _the software architect doesn't get to control any of it._ Start a new app or thread and it silently steals cycles from everything already running. The system is a commons, and it gets grazed.

This also gave us the concept of "thread safety" — an entire category of bugs that exists solely because multiple threads of execution share state on a single CPU. Mutexes, semaphores, lock-free data structures, memory barriers. Enormous bodies of engineering work, all devoted to managing a problem that was manufactured by the economic constraints of 1960.

The cleaner model would have been obvious to anyone not constrained by cost: put a zillion CPUs on a motherboard, each running exactly one thread, giving that thread 100% of its CPU. No scheduling. No preemption. No thread safety. The complexity just... evaporates.

CPUs used to cost as much as a house. So we wrote decades of software around the assumption that we'd only ever have one.

---

## What PBP Is About

That assumption is no longer true.

CPUs are cheap. Cores are cheap. The economics that forced the complexity of traditional operating systems onto us have reversed. And yet we keep writing software as if we're still rationing a single VAX.

PBP is an attempt to ease into doing things differently. Each part in a PBP diagram is conceptually its own agent, processing its own message queue, doing its own work. The coroutining kernel lets you run this model efficiently on today's hardware while the architecture remains clean: explicit hierarchy, explicit message passing, no shared mutable state leaking between parts.

The kernel is simple because the model is simple. A registry, and coroutining. That's all you need to build things that actually reflect how you think about them — rather than how a 1960s time-sharing system happened to organize them.

We didn't have to make software this complicated. We can stop.
