A transmogrifier is a text-to-text transpiler.

It is a tool, like a compiler.

When you build or maintain a compiler, you work in the maintenance directory and run tests that specifically test the operation of the compiler.

When you use a compiler, you work in a project directory and invoke the compiler on one of the files in your project. Any tests you run are specific to your project, not the compiler.

Typically, a compiler is released by moving it onto $PATH.

If the compiler needs to access files from its own maintenance directory, then something special needs to be done to give access to those files while working on files in the user's project.

I'm building transmogrifiers by using PBP and T2T which requires access to transmogrifier-specific grammar and rewrite files (along with other scripts and pieces).

To allow this to happen, I use two environment variables for the different paths:
- ${TARGET} is the name of a file in a user's project
- ${TRANSMOGRIFIER} is the name of a file in the transmogrifier's build directory.

In a type 2 usage, you use the tool to transmogrify a ${TARGET} file to some other file in your project directory. In the process the transmogrifier itself refers to its own files using ${TRANSMOGRIFER} while working on ${TARGET}. 

This approach is necessary because the transmogrifier consists of many pieces instead of being a single binary. Apparently, `npm` does this kind of thing also using `package` files
