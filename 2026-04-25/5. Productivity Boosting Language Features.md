- "compile" is "eval", except that it's not built into the programming language
	- if we consider the whole workflow for building code, the "compile" is "eval" in that workflow
	- "eval" (aka "compile") should not be built into the machine code released to users
	- Lisp, Javascript, etc. should not allow "eval" at runtime
		- but, using "eval" at development time is a useful tool
		- "eval" (aka "compile") can be a footgun, but it can also be a development productivity booster
		- if developers want to eliminate "eval" at development time, they should use a *linter* tool to warn/raise-errors about places in development code where "eval" still remains

- concurrency is not parallelism
	- on a single computer, we only *simulate* parallelism by making it appear that code is running concurrently, when in fact, it is using time-slicing to mete out small amounts of computer time to each process
		- software runs slower if an operating system is used and if threads are used
	- the current model for inter-process communication is memory-sharing, inherited from 20th century time-sharing techniques
		- this model carries a lot of baggage with it, like "thread safety" (which is only a problem if memory sharing is allowed)
		- this model is too complicated to extend into a future that uses distributed computers across large distances, i.e. cpus that are not on the same silicon chip
	- we can - easily - simulate parallelism on a development machine by using totally isolated software components that cannot share memory
		- Erlang apparently does this kind of thing, but, still allows a process to refer directly to other processes - by name (or by id)
		- PBP implements "topology blindness" by specifying that processes can only refer to their own output queues, instead of referring to functions and innards of other processes
			- this is easy to implement even using modern programming languages 
				- you just need to create an output queue for each component, and you have to "lint" that only component-local functions are called and output data is sent only to the component's own output queue

- breaking out of synchronous mindset
	- components communicate only by passing data via queues
	- no return value is expected
		- a "send" is a fire-and-forget operation, the sender does not block waiting for a response
		- this can be implemented easily using modern programming languages
			- similar to "green threads" in operating systems
			- a distinguished routine - the Dispatcher - invokes components that have non-empty input queues
				- basically, the Dispatcher keeps a list of all components (first class functions, closures)
			- synchronization, if needed, can be explicitly specified by request/ack messages instead of implicit memory sharing (and all of the baggage that brings into the picture)
	- code libraries must be reactive components that receive input data via message sends
		- instead of relying on the functional paradigm and expecting to be "called" in a synchronous manner
	- declaring certain functions to be "async" is a low level approach to the issue, much like using GOTO - it works, but it is unstructured
	- we need structuring of message sending - kind of like "GOTO considered harmful" for messages
		- scalable layers
		- one kind of recursive component that composes wires between direct children
		- another kind of Leaf part that just does something, e.g can be implemented as a synchronous component (aka "async" function(s)) (same external appearance - one input queue and one output queue)
