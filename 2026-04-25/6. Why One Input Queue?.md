One queue per part (component) means that messages can be queued in order of arrival.

This means that we can "reason about" message timing and ordering.

This is how state machines work.

Each Part has multiple input ports.

Modern thinking about multiple inputs is to provide one queue per port, but this obfuscates timing and ordering information.

It is better to have only *one* input queue per part where all arriving messages are enqueued in order of arrival.

This means that messages need to be tagged with a port id.

I call this kind of message a *mevent* (for message-event).

# mevent structure
1. port tag (a string is good enough, a counted string (instead of C-like null-terminated string) adds efficiency)
2. payload (a blob of binary data)

The payload can be destructured by the receiving Part. 

We cannot use predefined data structures, since this would imply the need for a centralized authority of data shape (like ".h" files in C). 

The receiver must unpack data at runtime, and, the sender must know to send the correct data.

Note that, with advances in PEG technology it is possible to build little parsers into Parts, to parse the incoming data mevents.

Note that Scott Warren's "protocol objects" use LLMs to decide on the meaning of incoming data. This is, in effect, a "parse" of the data.

Aside: T-receptors in biology might give us ideas on how else to understand non-centralized payload formats.
