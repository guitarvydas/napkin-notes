The goal of projectional editing is o store enough syntactic and semantic information to allow reconstructing the source from machine-readable form to human-readable form, where human-readable form can take numerous different shapes.

Lisp appears to define a very low level structure for the above kind of information. Everything is normalized down to being a function.

Source code control only needs to save information in a machine-readable form. When a piece of code is checked out, a routine can run to reconstruct the information in a more readable form. We didn't think this way in the 20th century because computers were too slow and we didn't want to spend CPU cycles for tooling. Today, the tables have turned, we can use computers to enhance program development workflow(s) that were deemed impossible in the last century.

There is no rule that says that source code control apps must store source "as is" in some static, human readable form. They can store/retrieve code while applying transformations on the way in and on the way out. We can use extra CPU cycles to enhance our development workflow.
