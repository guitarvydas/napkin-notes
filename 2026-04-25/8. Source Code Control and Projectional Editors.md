The triple: `relation(subject, object)`.

Consider encoding the fact that variable `x` is an integer with value `42`:

```prolog
type(x, int).
value(x, 42).
```

This is identical in meaning to:

```prolog
value(x, 42).
type(x, int).
```

Order doesn't matter. There are no pointers. There is no tree to traverse. There is only a set of atomic facts and an inference engine that can assemble them into whatever structure a query demands.

From those two facts, inference can reconstruct:

```
var x: int = 42
```

The concise, human-readable form is _derived_, not stored. The stored form is the flat factbase. The readable form is a rendering.

This is exactly the pattern a source code control system could exploit. Store the factbase. Render on checkout. The rendering is a query over the facts — a transformation, not a retrieval.

**What Factbases Buy Over Lisp**

Lisp's s-expressions preserve tree structure. The parent-child relationship is encoded in nesting. To change the structure, you change the nesting.

A factbase has no nesting. Structure is inferred, not stored. This means:

- Adding a new relation to an existing subject requires no restructuring. You append a fact.
- Querying across subjects requires no pointer traversal. You pattern-match.
- Diffing two factbases is a set operation — which facts were added, which removed — not a tree diff.

A semantic diff of two versions of a program becomes: what facts changed? Not: what characters changed. Not even: what AST nodes changed. What _relations_ changed.

That is a diff at the level of meaning.

**The CPU Trade**

Prolog infers. Inference costs CPU cycles.

In 1980, this cost was prohibitive for routine tooling. Today it is not. A factbase representing a large codebase is still a small dataset by modern standards. The inference required to reconstruct human-readable form from a factbase is well within what current hardware makes trivial.

We are not proposing anything computationally exotic. We are proposing that tools run a Prolog query on checkout instead of copying bytes.

Nils Holm wrote a complete Prolog inferencer in Scheme and explained it in approximately six slides worth of code — "Prolog Control in Six Slides" (https://www.t3x.org/prolog6/). The point is not the language. The point is the size: a working inferencer consists of a small amount of code.

The Javascript port includes a diary of the porting process. These ports exist not to produce production inference engines but to demonstrate that the mechanism is simple enough to carry across languages by hand.

An inferencer can be written in any language. Prolog syntax is convenient — relations expressed as `relation(subject, object)` read naturally and the unification semantics are clean — but the syntax is separable from the mechanism. If Prolog syntax is an obstacle, replace it. The inferencing still works.

**MiniKanren: A Different Trade**

MiniKanren (minikanren.org) is a family of inference systems that make a different computational trade than Prolog.

Prolog expends CPU cycles for backtracking — it searches a space of possible solutions by trying, failing, and retrying. MiniKanren avoids backtracking by expending space instead. It represents the search space explicitly and explores it without the overhead of backtrack-and-retry.

Both achieve inferencing. They make different trade-offs between CPU and memory. In the context of source code control — where the factbases are relatively small and queries run at checkout time, not in inner loops — the trade-off is largely academic. Either approach works.

The practical point is: inferencing is a solved problem, it is small enough to port by hand, and the implementation language does not matter. A source code control system that stores factbases and runs inference on checkout is not proposing exotic machinery. It is proposing ordinary programming with a different data model.

In 1980, this cost was prohibitive for routine tooling. Today it is not. A factbase representing a large codebase is still a small dataset by modern standards. The inference required to reconstruct human-readable form from a factbase is well within what current hardware makes trivial.

We are not proposing anything computationally exotic. We are proposing that tools run a Prolog query on checkout instead of copying bytes.

**Inferencing Is Not Prolog-Specific**

Prolog is a convenient syntax for expressing inferencing. It is not the only way to do it.

Nils Holm wrote a complete Prolog inferencer in Scheme and explained it in approximately six slides worth of code — "Prolog Control in Six Slides" (https://www.t3x.org/prolog6/). The point is not the language. The point is the size: a working inferencer consists of a small amount of code.

I ported Holm's inferencer manually to Common Lisp (https://github.com/guitarvydas/cl-holm-prolog) and again to Javascript using OhmJS (https://github.com/guitarvydas/js-prolog). The Javascript port includes a diary of the porting process. An inferencer can be written in any language. Prolog syntax is convenient — relations expressed as `relation(subject, object)` read naturally and the unification semantics are clean — but the syntax is separable from the mechanism. If Prolog syntax is an obstacle, replace it. The inferencing still works.

**MiniKanren: A Different Trade**

MiniKanren (minikanren.org) is a family of inference systems that make a different computational trade than Prolog.

Prolog expends CPU cycles for backtracking — it searches a space of possible solutions by trying, failing, and retrying. MiniKanren avoids backtracking by expending space instead. It represents the search space explicitly and explores it without the overhead of backtrack-and-retry.

Both achieve inferencing. They make different trade-offs between CPU and memory. In the context of source code control — where the factbases are relatively small and queries run at checkout time, not in inner loops — the trade-off is largely academic. Either approach works.

The practical point is: inferencing is a solved problem, it is small enough to port by hand, and the implementation language does not matter. A source code control system that stores factbases and runs inference on checkout is not proposing exotic machinery. It is proposing ordinary programming with a different data model.

**Transformations on the Way In and Out**

There is no rule that says a version control system must store source verbatim.

A VCS can apply a transformation pipeline on check-in: parse the source, extract the semantic structure, store a normalized form. On checkout, the inverse pipeline runs: take the normalized form, render it into whatever human-readable shape the developer needs.

Those two shapes need not be identical. A junior developer might get a more verbose, annotated rendering. A senior developer might get a compact, dense form. A code reviewer might get a diff at the semantic level rather than the text level — showing what _changed_ structurally, not just what characters differ.

None of this requires storing anything extra. It just requires running a transformation on checkout.

**Why We Didn't Think This Way**

CPU cycles were the constraint. In 1975, spending processor time on pretty-printing or semantic reconstruction at checkout was unacceptable overhead. Tools had to be fast. Fast meant: don't transform, just copy bytes.

That decision calcified into assumption. The assumption calcified into convention. The convention calcified into "the way source control works."

We designed around a bottleneck that no longer exists.

**What Becomes Possible**

Once check-in and checkout are transformation boundaries, several things open up:

Semantic diffing. Compare ASTs or factbases, not character streams. Detect that a function was renamed even if it moved to a different file. Know which _relations_ changed, not which lines changed.

Multiple renderings of the same codebase. One stored form, many views. Render for documentation, for review, for onboarding, for debugging — each optimized for its purpose.

Notation independence. Store the semantic graph or factbase. Render into Python syntax, or Lisp syntax, or a diagram, or a table. The notation becomes a view over the model, not the model itself.

Programmatic normalization. Formatting debates disappear. The canonical form is the semantic structure. Human-readable renderings are generated, not hand-maintained.

**The Deeper Point**

We built 20th-century tools under 20th-century constraints. We then treated those tools as definitions of what source code control _is_, rather than as solutions to problems that no longer exist in the same form.

Source code control is not "store text files and track diffs." That's an implementation that made sense under specific historical conditions.

Source code control is "preserve enough information to reconstruct the program." How we store that information, and what form we present it in, are engineering choices — not axioms.

The computers are fast enough now. The question is whether we're willing to rethink the tools.

**Source Code Control Doesn't Need to Store Source Code**

**TL;DR**

Source code control only needs to store enough syntactic and semantic information to reconstruct human-readable form on checkout. We assumed otherwise because 20th-century CPUs were too slow to make transformation practical. That assumption is now obsolete.

---

**L;R**

**The Assumption We Never Questioned**

Source code control systems store text files. That's the unexamined assumption baked into every tool from SCCS to Git. The text file is treated as the ground truth. What you check in is what gets stored. What gets stored is what you check out.

This assumption was born of necessity. In the 20th century, CPU cycles were expensive. Spending them on tooling was a luxury most shops couldn't afford. So we stored source "as is" — static, human-readable, character-for-character identical on the way in and out.

That constraint no longer exists. We haven't updated our tools to reflect that.

**What Source Code Control Actually Needs to Do**

Source code control has one job: preserve enough syntactic and semantic information to reconstruct the source faithfully.

It does not need to preserve the exact text representation. It needs to preserve the _meaning_ and _structure_. The human-readable form is a _rendering_ of that structure — not the structure itself.

This is the insight behind projectional editing. A projectional editor stores an AST or semantic graph, then renders it into whatever human-readable form is appropriate for the current task. The stored form and the displayed form are decoupled.

Source code control can work the same way. Check-in applies a transformation: source text → normalized machine-readable form. Checkout applies the inverse: machine-readable form → source text (or any other rendering).

**Lisp Pointed at This Decades Ago**

Lisp is worth examining here. A Lisp program is already close to its own AST. Everything normalizes down to either a function application or an atomic object. The s-expression syntax is a nearly direct serialization of the semantic tree.

This is why Lisp macros are powerful: the code _is_ the data structure. Transformations over code are just transformations over lists.

Lisp didn't go far enough — it still stores and transmits s-expressions as text — but it identified the right level of abstraction. The canonical form should be the semantic structure, not a particular textual rendering of it.

**Factbases: Flatter Than Lisp**

Lisp normalizes to a recursive structure. There is a deeper normalization.

The triple: `relation(subject, object)`.

This fragments meaning into atomic elements that carry no embedded structure. The elements can appear in any order. No element points to another in the low-level sense. A subject is an identifier — a name. An object is an identifier. Neither is a hard-coded address. Neither encodes position. A reference in a factbase is an indirection, not a pointer.

Prolog works with this representation. Prolog expends CPU cycles to infer structure from it.

Consider encoding the fact that variable `x` is an integer with value `42`:

```prolog
type(x, int).
value(x, 42).
```

This is identical in meaning to:

```prolog
value(x, 42).
type(x, int).
```

Order doesn't matter. There are no pointers. There is no tree to traverse. There is only a set of atomic facts and an inference engine that can assemble them into whatever structure a query demands.

From those two facts, inference can reconstruct:

```
var x: int = 42
```

The concise, human-readable form is _derived_, not stored. The stored form is the flat factbase. The readable form is a rendering.

This is exactly the pattern a source code control system could exploit. Store the factbase. Render on checkout. The rendering is a query over the facts — a transformation, not a retrieval.

**What Factbases Buy Over Lisp**

Lisp's s-expressions preserve tree structure. The parent-child relationship is encoded in nesting. To change the structure, you change the nesting.

A factbase has no nesting. Structure is inferred, not stored. This means:

- Adding a new relation to an existing subject requires no restructuring. You append a fact.
- Querying across subjects requires no pointer traversal. You pattern-match.
- Diffing two factbases is a set operation — which facts were added, which removed — not a tree diff.

There is a further practical consequence. Because facts are order-independent, a factbase can be stored in sorted form. Sorting does not scramble the semantic information — any ordering retains the full semantics. But a canonically sorted factbase is far more amenable to existing diff and git tooling than raw source text. Two versions of a program that differ in one relation will produce a clean, minimal diff over their sorted factbases. The same change in source text might produce a noisy diff spanning multiple lines, or no diff at all if the change was a refactor that preserved textual structure while altering meaning. Sorting is free. The improvement to diff quality is not.

This means the proposal does not require replacing git. Git can remain exactly as it is. What changes is what git is given to store. Feed git a canonically sorted factbase instead of raw source text, and git's existing diff, merge, and history machinery works better than it ever did on source. The transformation happens at the boundary — on check-in and checkout — and git operates on the normalized form without needing to know anything changed.

A semantic diff of two versions of a program becomes: what facts changed? Not: what characters changed. Not even: what AST nodes changed. What _relations_ changed.

That is a diff at the level of meaning.
