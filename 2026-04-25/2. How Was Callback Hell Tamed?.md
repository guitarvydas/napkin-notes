*how was callback hell tamed?*

Callback hell — the pyramid of doom where deeply nested callbacks made async code unreadable and error-prone — was tamed in several stages:

**1. Named functions (partial mitigation)** Instead of anonymous inline callbacks, you extracted named functions. This flattened the nesting visually but didn't solve the fundamental control-flow problem.

**2. Promises (ES2015)** Promises gave async operations a chainable return value. Instead of nesting, you chained `.then()` calls, turning a pyramid into a flat sequence. Error handling centralized into `.catch()`. Libraries like Q and Bluebird popularized this before it became standard.

**3. async/await (ES2017)** The real killer blow. `async/await` is syntactic sugar over Promises that lets you write async code that _reads like synchronous code_. The runtime suspends the coroutine at each `await` and resumes it when the Promise resolves. Error handling reverted to ordinary `try/catch`. This is the form most code uses today.

**4. Generator-based coroutines (parallel path)** Libraries like `co` (used heavily with Koa v1) used ES6 generators (`function*` / `yield`) to achieve a similar suspension/resumption model before `async/await` landed. This was a stepping stone that proved the concept.

**5. Reactive streams (RxJS et al.)** For _sequences_ of async events rather than single values, reactive extensions modeled event streams as composable pipelines with operators like `map`, `filter`, `merge`, `debounce`. This didn't replace Promises but addressed a different problem shape — continuous event flows rather than one-shot async calls.

---

**The underlying insight** is that callbacks inverted control: _you_ handed your continuation to someone else. Promises and async/await restored normal control flow by making the async result a _value you could hold_ and sequence on your own terms — which is really just the language catching up to what Continuation-Passing Style theorists had understood for decades.

From a PBP perspective, you'd note that all of this is still fundamentally _synchronous thinking_ dressed up — a linear chain of steps with a scheduler hiding underneath. True asynchrony (your 0D components, message-passing, no shared stack) sidesteps the problem entirely because there's no call stack to corrupt in the first place.
