# Mapping Software Notation Onto Hardware
2026-06-25

I seem to be straddling two extreme ends of the programming spectrum.

1. I like the idea of inventing and exploring custom notations that are "spherical cows" that avoid parts of the problem in favour of allowing deep thought about specific aspects of the problem of "programming".
2. I like to think in terms of how this stuff gets mapped onto hardware. This seems to conflict with the point above. What's called "programming" is - to me - only a slice of the programming puzzle. Our current languages - most derived from C and function calling conventions - conflict with efficient use of the hardware.

I think of Functional Programming as a notation that works well on paper, but maps onto hardware in a way that creates complexity for any problem domain beyond straight computation. The manifestation of FP on hardware is a useful tool, but should not be deemed to be the only way to program hardware. In general, it is a mistake to create programming languages that use only functions, even if those languages are not fully FP (i.e. C and most languages invented since then). Other forms of programming, like "systems programming", are ill-served by languages that are based primarily on the use of "functions", i.e. just about any modern programming language.

A CPU IC is just a building material, yet a majority of popular programming languages target only that kind of IC and borrow ideas from idealized Functional Programming. 

- mathematicians found this to be an inspiration for discretizing their mathematical thinking and notation
- they have been creating devices to help think about and concretely test this new found, more detailed notation
- they infer that this is a two-way street - that programming devices to automate their notation is also resulting in the only way to program these devices and is the only use-case for such devices
- it's like saying that wood can only be used to build dog houses, ignoring the fact that wood can be used to frame houses for humans or to build fences or to build soapbox derby cars for children, etc.

- centralized approach - program CPU to command all other non-CPU hardware

The first dis-efficiency came in when C defined _all_ subroutines to be "functions". I used to think that this was a good idea, since I'd been fed false advertising for many decades making me believe that C is "close to the hardware". K&R C _was_ close to the hardware it was aimed at (PDP-7), but when extrapolated to other CPU architectures, it was false advertising. This created the notion that C should be ported to all sorts of architectures and that we should adopt universal "calling conventions". This seems to have cemented a function-based approach into all modern languages. It is entirely possible to build languages that don't use the same "calling conventions" that we've adopted. For example, Forth - you can't build a real Forth using C or just about any other modern programming language. You _can_ build a real Forth using assembler on top of a CPU. You _can_ build something that *looks* like Forth on top of an anti-Forth substrate that uses common "calling conventions". As I understand it, Forth was originally built in COBOL using subroutines instead of using "functions". I believe that our function-based belief structure has resulted in accidental complexities, like adding code to support "functions" then adding more code to optimize away that previously added code. We, also, seem to believe that "operating systems" are laws of nature. In fact, operating systems are there only to support the function-based paradigm on CPUs, i.e. context-switching to allow multiple functions to share the same CPU, and code libraries that must be written in a function-based style.

Part of the false advertising is due to the fact that later CPU architectures, like the PDP-11, have `++` and `--` operators, which C directly incorporates. What was not pointed out was that C on PDP-11s and other CPUs needed extra code to implement the calling convention for "functions" (I've begun calling "functions" PVR-subroutines (Parameterized, Value-Returning Subroutines)).

This belief structure has caused decades of deep diving into hardware and software optimizations for optimizing the paradigm and undoing the baggage caused by the mapping of function-based concepts onto hardware.

It appears to me that people confuse the theoretical concept of "concurrency" with the practical concept of "parallelism". Concurrency is not parallelism. Concurrency is a notational aspect that works cleanly on paper, but maps to time-sharing on hardware and to further extra complexity on distributed hardware. Time-sharing and the use of operating systems means that apps run more slowly than would be possible. Time-sharing means that an app does not get to use 100% of a CPU. Operating systems spend CPU cycles and memory footprint dealing issues that are often not needed by most apps. Time-sharing was a way to simulate distributed computers using only one CPU, but, we don't need to worry about that any more.

The following articles discuss this issue some more [Concurrency (Part 1 of 2)](https://programmingsimplicity.substack.com/p/concurrency-part-1-of-2?r=1egdky) and [Concurrency (Part 2 of 2)](https://programmingsimplicity.substack.com/p/concurrency-part-2-of-2?r=1egdky).

I contend that using a different substrate paradigm - Parts Based Programming (PBP) - built on top of synchronous, sequential CPUs, could give better performance and ability to reason about things that the function-based paradigm avoids, e.g. side-effects, state, etc. These issues are not fundamentally bad, they are just bad if you stick to the function-based paradigm or to the functional programming paradigm.

Another inconvenient issue is that we strive for a unified programming model. Function-based programming cannot solve *all* issues conveniently, PBP cannot solve *all* issues conveniently, [Statecharts](https://guitarvydas.github.io/2023/11/27/Statecharts-Papers-We-Love-Video.html) cannot solve *all* issues conveniently. We should, instead, be striving for ways to combine many different paradigms and laser-focussed languages into complete solutions for whole problems. We should not force ourselves to adopt a substrate paradigm based only on the use of functions - this warps our ability to create concise solutions to other kinds of sub-problems.
