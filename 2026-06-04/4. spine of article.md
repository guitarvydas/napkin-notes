It mostly feels right, except that the part about function calls might need more nuance.

> **Function calls as centralization mechanism:** The call stack. Shared memory. The scheduler. All centralization mechanisms imposed on decentralized hardware.

The Functional paradigm is used in mathematics. Early approaches to handling this new medium failed to recognize it as a new medium in need of new notation. It was believed that mathematical notation was "good enough" to describe the operation of these new reprogrammable machines and other options for notation were swept aside, even though these new machines were capable of doing time-related things, something which is clearly outside of the sweet spot of the functional paradigm. We have been backing into gotchas due to this approach for decades. We can't see the forest for the trees. We've been pushing functionalism onto RMs (Reprogrammable Machines) and are now making trade-offs in the belief that RMs are functional paradigm machines. Each time we hit a use-case outside of the sweet spot of the functional paradigm, we invent new bandaids and move on. For example, we discovered that graphics processing could use more optimization, so we just did more of the same - we created wider register functional-paradigm machines to handle parallel calculations and called it SIMD. Then, we found that we could do even better by adding more loose coupling and called that SIMT. Each design iteration becomes more tortured and more complicated and harder to comprehend, hence, harder to get correct. Then, we discovered that we could kludge these tortured architectures to support LLM progress. 

Shared memory is an optimization for speed that makes data-passing (message-passing) faster due to shorter communication paths. Internet-y systems composed of many parallel RMs do not have short communication paths - they do have multiple paths, but, we've been ignoring that issue because our centralized mental model let us get away with the single happy-path concept. 

The scheduler is necessary as scaffolding to allow a program to be composed of many single-threads (tightly-coupled chains of functions calling other functions). Maybe we wouldn't need this kind of scheduler if we were to use a different notation?

The call stack is provided by hardware CPUs to allow memory compaction. The same code can be cut/pasted into only one place and called - synchronously - from many different places. We were able to fold code and save memory. Then we invented compiler optimizations ("inline") to unfold code to speed things up at the cost of duplicating code in memory. The callstack  was a reasonable idea when memory was expensive.
