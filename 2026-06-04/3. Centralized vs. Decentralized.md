
The parallel printer cable tells the whole story. Twenty-five conductors. The host computer drove every signal. The printer waited to be told what to do, byte by byte, in lockstep. Control was centralized in the host. The LaserWriter cable had three pins. The Mac sent a PostScript program and moved on. The printer ran its own CPU, made its own decisions, produced its own output. Control was distributed. Neither machine was in charge of the other.

This is the difference between centralized and decentralized control. Function calls are centralized. The caller owns the callee. The callee cannot proceed until called. The caller cannot proceed until the callee returns. Everything flows through a single point of control. Message passing is decentralized. The sender sends. The receiver receives. Neither waits for the other. Neither owns the other. The Mac did not own the LaserWriter. It sent a message. What happened next was the LaserWriter's business.

Modern programming languages are designed for centralized control. The call stack is a centralization mechanism. The operating system scheduler is a centralization mechanism. Shared memory is a centralization mechanism. Every thread reaching into the same heap is asking for permission from a single authority. We built decentralized hardware — multiple CPUs, distributed networks, independent devices — and then wrote software that treats all of it as if one thing is in charge.

**Standard parallel printer (Centronics):** 36-pin connector on the printer, 25-pin DB-25 on the computer, 25 conductors. Bulky, stiff, thick.

**Apple LaserWriter (PostScript/LocalTalk):** 3-pin mini-DIN connector, RS-422 serial, thin cable. Later versions used PhoneNet — literally standard telephone cable with RJ-11 plugs.

But the deeper point isn't the cable. The LaserWriter was a message-passing machine. The Mac sent a PostScript _program_ to the printer. The printer — running a Motorola 68000 at 12 MHz, faster than the Mac itself — received that program and executed it independently. The Mac didn't control the printer step by step. It handed off a message and moved on. That's the 0D/PBP model in hardware from 1985.

The standard parallel printer was the opposite: a dumb device controlled byte-by-byte by the host, tightly coupled, synchronous, the computer doing all the work.

The thin cable wasn't an accident. Fewer wires reflected looser coupling. The architecture determined the connector.
