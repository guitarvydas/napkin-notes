# One Language To Rule Them All

**TL;DR** The functional paradigm is not general-purpose. It is a specialized notation aimed squarely at the domain of classical physics and mathematics. We mistook it for a foundation and spent decades bolting onto it everything it was not designed to handle. The result is bloat — measured in orders of magnitude — and a habit of removing things that worked, replacing them with things that didn't, and then quietly re-introducing the originals under new names. We never stopped to measure whether any of this was worth it.

---

**L;R**

SICP — the Structure and Interpretation of Computer Programs — is scaffolding for SICM, the Structure and Interpretation of Classical Mechanics. SICM uses a programming notation to express classical mechanics precisely. SICP builds the notation that SICM uses. Operating systems, in turn, are scaffolding for SICP — they provide the environment in which functional programs run. Each layer is well-engineered for a narrow purpose. None of this is general-purpose programming. It is a stack of specialized tools, each aimed at the same target: mathematical, functional, deterministic computation.

This would be fine if we had recognized it for what it is. We did not. We took the functional paradigm and declared it the foundation of all programming. As soon as C made it standard practice for every subroutine to use a functional calling convention, we stopped noticing that other conventions existed and were valid. The paradigm we had chosen for one class of problems became the only paradigm we considered for any problem.

The functional paradigm has a precise implicit target: classical physics. In that domain, it is elegant and powerful. Equations are timeless. Functions are pure. State does not exist. Time does not pass. The notation fits the domain exactly because the domain was defined to exclude everything the notation cannot handle. This is not a criticism. It is a description of what the notation is for.

The problems began when we tried to stretch the notation beyond its target. Re-assignment was the first major addition. In a pure functional system, a variable holds one value. In the real world, things change. Rather than reaching for a different notation suited to expressing change — state machines, for instance — we bolted re-assignment onto the functional paradigm. The notation became less clean. The reasoning guarantees it had offered began to erode. We did not get a more general tool. We got a worse functional tool with some state stapled to the side.

Dynamic binding followed a similar arc. Early Lisp used dynamic binding — variables resolved at runtime based on the call environment. This was later judged to be unpredictable and replaced with static scoping, which resolves variables at compile time. Static scoping enabled a class of compile-time analysis that dynamic binding made difficult. The trade seemed worthwhile. Decades later, we introduced JIT compilation — just-in-time, runtime compilation — which reintroduces exactly the kind of runtime dynamism that static scoping was designed to eliminate. We removed dynamism, spent years building analysis tools that depended on its absence, and then put it back. The loop cost us decades of work and produced no net gain over where Lisp 1.5 started.

Lisp 1.5, in 1962, did the core job in approximately 27 kilobytes. It had garbage collection, recursion, and a runtime evaluator. It was powerful enough to express a proof assistant. VisiCalc, in 1979, invented the spreadsheet paradigm in 20 kilobytes. The first edition of Unix, in 1971, ran the entire operating system — kernel, shell, tools — in 24 kilobytes of total memory. Turbo Pascal 1.0, in 1983, delivered a complete integrated development environment — editor, compiler, linker, and all libraries — in a single 38-kilobyte file. These were not stripped-down toys. They were world-changing tools.

Today we speak in gigabytes. The Linux kernel alone exceeds 30 million lines of source code. A modern language runtime often requires hundreds of megabytes before a user program loads. We grew the footprint by four to five orders of magnitude. We did not stop to ask whether we were getting four to five orders of magnitude of additional value. We assumed we were.

Sector Lisp answers the question. Sector Lisp is a modern implementation of Lisp with garbage collection, written in 223 lines of assembly. Its footprint is 436 bytes. It is powerful enough to run the proof assistant that shipped with Lisp 1.5. The difference between 27 kilobytes in 1962 and 436 bytes in 2021 is not hardware improvement. It is the removal of everything that was added in the attempt to make Lisp general-purpose. Sector Lisp does not allow re-assignment. It does not use a heap in the conventional sense. It does exactly what the pure functional paradigm does, nothing more, and it fits in less than half a kilobyte. The bloat was the generalization. The core was always small.

Type checking is the most visible symptom of this disease. We have spent decades elaborating type systems, adding annotations, refining inference engines, and debating the merits of different type disciplines. The argument for all of this effort is that type checking catches errors before they reach production. Robert Smallshire, presenting at InfoQ, has cited research suggesting type checking catches approximately 3 percent of real-world errors. Whether that number is precisely right or not, it points to a question we have not asked seriously: what did the other 97 percent cost us, and what were we doing before type checking existed?

Sketchpad, in 1963, pioneered the graphical user interface, object-oriented programming, and geometric constraints. VisiCalc transformed the personal computer from a hobbyist toy into a business tool. Unix built the foundation on which the internet runs. None of these were built with automated type checking. The creators of these systems were not reckless. They were working in notations suited to their domains, small enough to hold in their heads, expressive enough to say what they meant directly. Type checking, as it has grown, is largely a response to the complexity introduced by trying to use one large general-purpose language to express everything. The type system grows because the language grows because the problem domain keeps expanding beyond what the original notation was designed for. With multiple small, domain-specific notations, much of what the type system guards against would not exist in the first place.

This is not an argument against type checking. It is an argument that type checking is a symptom, not a solution. The solution is to stop forcing every problem into a notation that was designed for a different class of problems, and to build or borrow a notation that fits the problem at hand. Small notations can be held in the head. Small notations do not require elaborate annotation to remain consistent. Small notations, as Turbo Pascal and VisiCalc and Unix V1 demonstrated, can change the world in under 40 kilobytes.

We did not get here by making bad decisions. We got here by making locally reasonable decisions without measuring their cumulative cost. Each addition to the functional paradigm made sense in isolation. Re-assignment was useful. Static scoping had real benefits. Type systems caught real bugs. But we kept adding and never subtracted, never benchmarked, never asked what the baseline was and whether we had improved on it. The baseline was 20 kilobytes and a spreadsheet that changed the world. We owe it to ourselves to compare honestly.
