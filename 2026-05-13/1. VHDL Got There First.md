**VHDL Got There First**

TL;DR Hardware engineers were forced to invent PBP. Software engineers were not forced, so they didn't.

---

L;R

Hardware description languages like VHDL exist because physics demanded it. You cannot fake concurrency in silicon. Every component runs simultaneously, communicates through ports, and has no idea what it is wired to. The topology is declared externally, in an architecture body, not inside the components themselves.

That is PBP.

VHDL processes communicate through signals attached to entity ports — not through shared state, not through function calls. A component does not call its neighbor. It emits on an output port and forgets. Someone else's input port receives it. The wiring is someone else's problem, declared at a higher level of hierarchy.

That is 0D.

A VHDL design hierarchy — a top-level entity instantiating sub-components — is a Container holding Leaves. The Container knows the topology. The Leaves do not.

Software has no such physical constraint. CALL/RETURN works fine on a single CPU. So software engineers never felt the pressure that hardware engineers felt, and they kept CALL/RETURN as the universal default, long past the point where distributed systems made it a lie.

VHDL falls short of PBP in one significant way: it is pseudo-synchronous. The simulation engine imposes delta cycles — a global ordering pass that serializes what looks concurrent. Real hardware escapes this via careful timing discipline. PBP escapes it by making the queue the only synchronization primitive and letting the scheduler be dumb.

The deeper point is that hardware engineers solved the concurrency problem in the 1980s, named it VHDL, and software engineers ignored it entirely.
