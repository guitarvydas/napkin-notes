# Five Whys — Video Script

---

## Scene 1: The Problem With Vague Problems

**Visual:** Animator draws a simple black box on screen — one arrow entering from the left, one leaving to the right, one small arrow leaving from the bottom labelled ✗.

**Narration:** Every programmer and every consultant eventually faces the same situation. A customer comes to you with a problem. You ask what they need. They give you an answer — and the answer isn't detailed enough to write a single line of code.

They know something is wrong. They just can't express it precisely enough for software to help them.

One technique for closing that gap is called the Five Whys. Each time you get an answer, you rephrase it as a new "why" question. You repeat five times. Each question is grounded in the previous answer rather than restating the original. By the fifth round, you typically have something concrete enough to act on.

We are going to build a tool that runs this process automatically, using an LLM — and we are going to build it using FDD, Feature-Driven Development, which works as a top-down / bottom-up spiral.

**Visual:** The black box sits on screen. A label appears: "Five Whys". A question mark enters from the left. A detailed, bulleted answer exits to the right.

---

## Scene 2: Starting at the Top

**Visual:** The top-level diagram sketch appears — one input port, one output port, one error port.

**Narration:** At the very top level, we know only this: one input, one output, and one error output. The two outputs are mutually exclusive — if an answer comes out, nothing comes out of the error port, and vice versa. There is also a noticeable delay between the input arriving and the answer appearing.

That is all we know. It is deliberately vague. FDD says: start with what you know, then alternate between the top and the bottom of the problem, chipping away at both ends.

**Visual:** The diagram is labelled. An arrow points inside the box with a question mark — "what's in here?"

---

## Scene 3: Diving to the Bottom — Getting the LLM Running

**Visual:** A terminal window opens. The `agency` repository appears in a file browser.

**Narration:** We know the box contains an LLM. So our first concrete task is to get an LLM running at all — not wired into anything, just working from the command line.

We borrow an existing LLM library called _agency_, written in Go. That choice of language is deliberate: it shows we are not locked into any single language or runtime at this stage. We install dependencies, build from source, and run a simple test.

**Visual:** The terminal shows:

```
$ ./@install
$ ./@make
go build examples/cli/main.go
echo "is concurrency considered difficult?" | ./agency/main -model gpt-4o-mini -maxTokens 1000 -temp=1 -prompt "concise responses"
Yes, concurrency can be difficult due to issues like race conditions, deadlocks, and the complexity of coordinating multiple threads or processes...
```

**Narration:** The LLM answers the question. We now have something concrete: a command line that reads from `stdin` and writes to `stdout`. That is exactly what we need to build a shell-out part.

---

## Scene 4: Wrapping the LLM as a Part

**Visual:** The drawio editor opens. The raw shell-out Leaf part appears — a box whose name is the command line prefixed with `:$` .

**Narration:** In PBP — Parts-Based Programming — a shell-out part is the simplest kind of Leaf. Its name is the command to run, prefixed by `:$` . Input arrives on `stdin`; output leaves on `stdout`. The PBP kernel handles the rest.

We wrap this Leaf inside a Container part called "LLM" to give it a clean, readable name. That Container becomes our first reusable part.

**Visual:** The LLM tab appears in the drawio file. The shell-out part sits inside it with its input and output ports. The tab is labelled "LLM".

**Narration:** Once a part goes on the palette, we stop thinking about it. The internals — Go code, the OpenAI API, command-line flags — are hidden. From now on, "LLM" is just a box with one input and one output.

---

## Scene 5: The First Test Jig

**Visual:** A second drawio tab appears: "LLM-test-jig". It shows a simple diamond (input) wired directly to the LLM box, wired to the output diamond.

**Narration:** Before we build anything more, we test what we have. The simplest possible test jig is a container that feeds its input directly to the LLM and passes the output out. We write a small Python coupler to drive it from the command line.

**Visual:** The `main.py` code appears briefly, then `@makec`. The terminal runs `./@make`.

**Visual:** Terminal output:

```
[
  {
    "": "Yes, concurrency can be difficult due to challenges such as race conditions, deadlocks, and ensuring data consistency across multiple threads or processes..."
  }
]
```

**Narration:** It works. The output is a JSON array of tagged strings — the port name as the key, the LLM's answer as the value.

Two things stand out immediately. First, the answer contains several distinct issues, not one. That means the rephrasing step will need to handle multiple issues at once. Second, the run takes a couple of seconds and costs money. If I am conscious of that cost, I may hesitate to run tests freely — which kills rapid iteration.

Both are cheap insights to have now, before we have built anything complicated.

---

## Scene 6: The Stub — Free Iteration

**Visual:** The LLM tab in drawio. The shell-out Leaf is replaced by a straight wire — input port connected directly to output port.

**Narration:** The fix for the cost problem is a stub. Instead of the shell-out part calling the API, we wire the input directly to the output. Any string sent in comes straight back out. The test jig is otherwise unchanged.

**Visual:** Terminal runs `./@make`. Output:

```
[
  {
    "": "is concurrency considered difficult?"
  }
]
```

**Narration:** Input in, same string out. No API call, no cost, instant results. We can now iterate as fast as we want. When we are ready to test with a real LLM, we swap the stub back — one change, in one place.

---

## Scene 7: Building the Counter

**Visual:** A new Python file — `counter.py` — appears in a code editor.

**Narration:** We need a part that counts incoming messages. The first four go to one output port, labelled `<5`. The fifth goes to a different output port, labelled `5th`. This controls how many times the "why" loop runs.

We write this as a Python Leaf. It is too simple to justify a diagram.

**Visual:** The `counter.py` code is visible: `class Counter`, `self.max = 5`, `self.count = 1`, the `handler` method routing to `"<5"` or `"5th"`.

**Visual:** A new drawio tab — `counter-test-jig` — shows the counter box wired between the LLM and the output.

---

## Scene 8: Probing the Feedback Loop

**Visual:** The `counter-test-jig` diagram, now with a small "intern" output port tapped off the `<5` output. A probe icon appears.

**Narration:** We want to watch what is happening inside the loop without changing its behaviour. PBP supports probe parts — small taps that print the data flowing past them, like an oscilloscope probe on a circuit. Because PBP uses fan-out, adding a probe means drawing one extra arrow. No code changes, no restructuring.

**Visual:** The terminal runs. Multiple "Info" lines appear, each showing the string passing through the counter. Then the final JSON output at the bottom.

**Visual:** Annotation circles highlight the probe messages in the terminal.

**Narration:** Probes are given internal priority, so they always print before the data moves further along the fan-out chain. The result is a clear, ordered view of what is happening, when.

We immediately spot two bugs. The counter has an off-by-one error — it counts to three instead of four. And the drawio "Formatted Text" setting is on, leaking HTML markup into the port names.

---

## Scene 9: Fixing the Bugs

**Visual:** Split screen. Left: drawio with the Text panel open, "Formatted Text" being unchecked. Right: `counter.py` with `self.inc()` being moved to the correct position.

**Narration:** Both fixes are trivial. We uncheck "Formatted Text" on all elements in the diagram, and we move the increment call so the counter advances at the right moment.

**Visual:** Terminal runs again. Clean output: four intern messages, then the final output. No markup, correct count.

---

## Scene 10: Adding the Rephrase Part

**Visual:** The drawio editor. A new tab appears — "Rephrase". Inside it: a string constant part, a String Concat part, and the Raw LLM part. Three probe parts are attached.

**Narration:** Now we wire in the rephrasing step. The Rephrase part prepends a canned instruction — "Rephrase each issue as a concise question; ask for concise answers" — to whatever the LLM just said, and sends the combined string back to Raw LLM. The result is the rephrased question set.

The output from Rephrase feeds back into "Initial Question" at the top level, completing the loop.

**Visual:** The `counter-test-jig` tab shows the full wiring: input → Initial Question → counter → Rephrase (feedback) → Initial Question again, with the `5th` output of counter going to the final output diamond.

**Narration:** Note that this is not recursion. There is a real time delay, and the feedback message goes to the end of Initial Question's input queue, not the front. The loop runs to completion at each step before the next message is processed.

---

## Scene 11: First Snag — the Orphan Port

**Visual:** The terminal. A warning message appears:

```
WARNING in Rephrase: Port "" is not inside a container but is used in connections. This connection will be ignored.
********* quit: suspicious drawing *********
```

**Narration:** Something is wrong. The PBP tool reports an unconnected port and quits.

The culprit is a drawio quirk. Drawio is a drawing editor, not a programming editor. It allowed a port to be drawn _near_ the String Concat box without being graphically _attached_ to it. To the eye it looks connected. To the tool it is not.

**Visual:** The Rephrase tab in drawio. The offending port is highlighted. It is dragged inward until the String Concat box turns purple — drawio's signal that the port is now truly contained within the shape.

**Narration:** The fix is a graphical gesture: drag the port inward until the parent shape highlights. Once the drawing editor is satisfied, the PBP semantics can be correctly read from the file.

---

## Scene 12: First Light — the Feedback Loop Runs

**Visual:** The terminal. A long stream of "Info" probe lines scrolls past.

**Narration:** After fixing the port, we run again. This time the output is much longer.

**Visual:** The terminal output is annotated with floating labels: "Loop 1", "Loop 2", "Loop 3", "Loop 4", "Final Output".

**Narration:** Loop 1 — the initial question enters. The LLM answers. The Rephrase part sees four issues but collapses them into a single question. We note this as something to improve.

Loop 2 — the single rephrased question re-enters. This time the LLM produces five numbered issues. Rephrase generates five distinct questions.

Loop 3 — five questions in. Five answers out. Five new questions rephrased.

Loop 4 — the questions sharpen further. The language grows more precise.

**Visual:** The final JSON output appears, clean and readable:

```
1. Race conditions occur when multiple processes or threads access shared
   resources simultaneously, leading to unpredictable outcomes...

2. Deadlocks happen when two or more processes are each waiting for the
   other to release resources, causing them to be stuck indefinitely...

3. Concurrency adds complexity by introducing issues like race conditions,
   deadlocks, and increased difficulty in managing shared state...
```

---

## Scene 13: Cleaning Up and Evaluating

**Visual:** The drawio tabs. Probe parts are deleted one by one — "Q", "A", "newQ", "again" — leaving only "rephrased". The diagram becomes visually clean.

**Narration:** The probes have done their job. We remove them. The machine regenerates all the intermediate results automatically — we pay nothing to re-run, because re-running is exactly what rapid iteration means.

**Visual:** A clean terminal run. Only the "rephrased" probe and the final output are visible.

**Narration:** The initial question was "is concurrency considered difficult?" — vague, answerable in a sentence.

The final output is three concrete, actionable points: race conditions, deadlocks, synchronization. As a programmer scoping a project, reasoning about those three things is far more productive than reasoning about "difficulty" in the abstract. The tool is working.

---

## Scene 14: What FDD Taught Us

**Visual:** The camera pulls back to show all the drawio tabs side by side: LLM-test-jig, LLM, counter-test-jig, Initial Question, Rephrase, Raw LLM.

**Narration:** Let's take stock of what the FDD process taught us.

**Visual:** The top-level sketch from Scene 2 reappears, then is overlaid with the final `counter-test-jig` diagram — the same basic shape, now filled in.

**Narration:** The spiral works. Every pass from the top — sketching the shape of the solution — was paired with a pass from the bottom — making one concrete part real. Neither direction alone is sufficient. Together they converge.

**Visual:** The LLM part is highlighted, then shown being used inside Initial Question and inside Rephrase simultaneously.

**Narration:** Parts behave like LEGO. Once a part is built and tested, it goes on the palette and we stop thinking about it. The LLM was wrapped once and reused twice — in Initial Question and in Rephrase — without modification or a second thought. The internals stayed hidden.

**Visual:** A probe part appears, gets connected, the terminal shows output, and then the probe is deleted. Repeat quickly.

**Narration:** Probes made bench-testing effortless. Insert a probe, run, observe, remove. No code changes, no restructuring. The fan-out mechanism is what makes this possible: tapping a signal is just drawing one extra arrow.

**Visual:** The original command-line LLM test from Scene 3 reappears alongside the final system diagram.

**Narration:** And finally: we never worried about performance, cost, or memory footprint while designing. Those concerns are real, but they belong to a later phase. The design is clean and the parts are independent — a production-engineering team could take this and optimise it without touching the architecture. Keeping those two phases separate is itself one of the most valuable things FDD demonstrates.

**Narration:** We started with a box and a vague question.

We ended with a working tool that makes vague questions less vague — automatically, iteratively, and cheaply.

That is FDD.
