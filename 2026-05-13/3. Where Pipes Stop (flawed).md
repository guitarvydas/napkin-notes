**Where Pipes Stop**

Unix pipes prove the concept but freeze it in one shape. A pipeline is a chain — one component feeds the next, left to right, single file. That topology is hardwired into the syntax.

PBP removes that constraint. A component can have multiple input ports and multiple output ports, each with a name. The wiring can fan out — one output feeding several components. It can fan in — several outputs feeding one input. It can loop back. The topology is not a chain; it is a graph, and the graph is declared by the Container, not implied by the syntax.

Unix pipes also stop at the process boundary. You cannot take a pipeline and drop it inside a program as a first-class component, give it ports, and wire it into a larger design the same way you wired its pieces together. The shell is not composable in that sense. PBP is composable all the way down — a Container of components is itself a component, with its own ports, indistinguishable from a Leaf from the outside.

The third limit is data. Pipes carry bytes. Everything upstream must serialize; everything downstream must parse. PBP messages carry structured data. The protocol is the component's responsibility, not the operating system's.

Pipes gave you the paradigm. PBP gives you the full thing.

PBP, also, defines "processes" in a simple way, that can be implemented efficiently at a lower, language level. Currently, I'm using the Python version of the PBP kernel. pipes carry bytes. PBP mevents are 2-part (1) a tag id and (2) a payload of (unstructured) bytes. Basically key/value pairs. This makes it possible to think in terms of multiple output / input ports -- which lets you think beyond the strict chain-of-processes paradigm. The key is the port, the value is the data. This makes it possible to implement parts with exactly one input queue which lets you reason about mevent ordering - something not natural in most programming languages (I think that the workaround is to use ad hoc state variables).
