# Impure Dataflow

Functions - the functional paradigm - when mapped onto physical hardware do not inherently implement pure dataflow. 

You need to add code to break the tight coupling of functions and to achieve pure dataflow, because hardware must obey the laws of physics and time and ordering, unlike functional notation written on paper which isn't affected by the laws of physics (functions can be replaced and expanded faster than the speed of light or in any order)

This is a precise and underappreciated point. The functional paradigm achieves its elegance by operating in a physics-free space. On paper, a function can be substituted for its result instantaneously and in any order — referential transparency is the formal name for this property, and it is genuinely useful for reasoning. But hardware exists in time. Electrons move at finite speed. Memory reads have latency. Two operations cannot literally occur simultaneously on the same physical resource. The laws of physics are non-negotiable.

**The mapping problem**

When you compile functional code to hardware, the runtime must secretly reintroduce all the time and ordering that the notation pretended didn't exist. The call stack enforces ordering. The scheduler enforces sequencing between "concurrent" operations. Locks and mutexes manage the physical reality that two threads cannot simultaneously write to the same memory location, a constraint that has no analogue in the mathematical model. None of this machinery is visible in the functional notation — it is injected by the compiler and runtime as a hidden layer between the clean mathematical abstraction and the physical reality underneath.

This hidden layer is not free. It costs cycles, it costs memory, and most importantly it costs comprehensibility — the programmer reasons about the functional model, but the bugs live in the gap between that model and the physical execution.

**Pure dataflow requires explicit design**

To achieve genuine pure dataflow on hardware — where components communicate only through messages, have no shared state, and are truly decoupled — you have to add code that the functional paradigm not only doesn't provide but actively works against. You must explicitly break the caller-callee coupling that CALL/RETURN imposes. You must explicitly manage queues. You must explicitly prevent the implicit synchrony that function calls enforce. Every async/await, every message queue, every actor framework, every reactive stream library is engineering effort spent dismantling the tight coupling that the functional notation built in by default and that the hardware dutifully implemented.

The irony is complete: the paradigm that claims to best model pure mathematical dataflow requires the most work to achieve actual dataflow on physical hardware. Whereas hardware itself — ICs, buses, signals, pins — already implements something much closer to genuine pure dataflow natively, and always has.

**The on-paper vs. on-hardware gap is the source of most accidental complexity**

What you're identifying is that the functional paradigm's purity is a notation-level property that does not survive compilation. The purity exists in the source code. The execution is thoroughly impure — stateful, ordered, sequential, time-bound. The programmer writes in one world and the machine executes in another, and the translation between them is where callback hell, race conditions, deadlocks, and the entire ecosystem of concurrency primitives come from. These are not hard problems inherent to computation. They are artifacts of the mismatch between a physics-free notation and a physics-bound substrate.
