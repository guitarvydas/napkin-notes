This is a compelling and well-reasoned position. A few thoughts:

**Where you're clearly right**

The "milquetoast union" critique is sharp. Most general-purpose languages end up with a dominant paradigm and then awkward bolt-ons for everything else — C++ adding templates and then lambdas, Python adding async, Java adding streams. The result is that the language _can_ express many things but expresses most of them poorly. The idiom for concurrency, for example, feels alien relative to the idiom for OOP in the same language.

Your point about control flow is particularly well-observed. Try-catch, async/await, callbacks, and promise chains are all textual approximations of what is fundamentally a _graph_ problem. A statechart (à la Harel) makes concurrency, parallelism, hierarchy, and multiple valid reactions to events completely natural — because the medium matches the concept. Forcing it into sequential text means you're constantly fighting the notation.

**The deeper principle at work**

What you're describing aligns with the idea that **notation shapes thought**. A language isn't just a way to instruct a machine — it's a cognitive tool. When the notation is ill-fitted to the problem, programmers spend mental energy on the translation, not the problem. Small, focused languages minimize that translation cost. This is part of why mathematicians don't write proofs in Python — mathematical notation _is_ a TPL, just a highly specialized one.

**Some productive tensions worth considering**

- **Interoperability cost.** Little languages multiply the seams between them. Those seams — marshalling data, synchronizing state, debugging across language boundaries — can become their own source of complexity. The gain in expressive clarity per language has to outweigh the friction at the interfaces.
- **Tooling gravity.** Big monolithic languages attract debuggers, profilers, IDEs, and communities. Little languages often don't, which can make them expressive in theory but painful in practice. This is arguably a solvable engineering problem rather than a fundamental objection, but it's a real one.
- **The DPL literacy gap.** Diagrammatic languages have historically struggled with version control, diffing, and text-based tooling — things the industry has spent decades optimizing for TPLs. Statecharts are a great example: conceptually superior for control flow, yet rarely used in mainstream practice partly because a `.scxml` file in a git diff is nearly unreadable.

**The most interesting implication of your view**

If you take it seriously, the right architecture for a programming _workflow_ is essentially a **composition of DSLs** — each one the best possible notation for its subdomain — with a thin integration layer. This is actually what some mature systems quietly do: a data pipeline might combine SQL, a YAML config DSL, a visual DAG editor, and Python for escape hatches. But it's rarely _designed_ that way from the start; it usually accumulates accidentally. Your framing suggests it should be a first-class design goal.
