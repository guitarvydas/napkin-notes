Big debuggers suffer from the single-paradigm model. They help you debug code written in one paradigm - with current debuggers, the sync/sequential paradigm - but are clumsy when dealing with other paradigms, e.g. async parallel units of code. Little, very focussed, languages can avoid the need for debugging by simply making the DI (Design Intent) more obvious and easier to examine. The sync/sequential paradigm makes it clumsy to deal with fan-out, but, fan-out makes it easier to express "probes" that you can insert into the code and just as easily remove from code. Breaking out of the sync/seq paradigm might open up new ways to view code and control flow and bugs. In fact, I've already experienced this with PBP - I can simply draw extra "probe" parts on the diagram which help me view the intermediate parts of the code in ways that are meaningful at a high level - examining PBP code using sync/seq debuggers is too low level and painful (akin to staring at core dumps).

REPL-based languages make it possible to debug a program without a formal debugger. You just write debugging/observational code in the same language and watch the output. In C, this was called "printf debugging" but was so weak - you couldn't use C to debug itself in an iterative manner - that it was frowned upon. Again, the sync/seq paradigm tended not to reflect the semantic higher design levels of the code being debugged, so it felt clumsy and inadequate.

Moldable Development, espoused by The Glamorous Toolkit (https://gtoolkit.com), IMO, is based on the observation that sync/seq debuggers are too low level. GT provides a way to provide different views on your code.

Projectional editors have the goal of providing different syntax skins for the same underlying semantics. As far as I can tell, the assumption is made that "code" must be written in textual form. We are much closer to that goal today due to the existence of tools like OhmJS. In my opinion, mature dev languages like Common Lisp provide a wide variety of tools with a very bare, normalized syntax, and, IMO, might form a basis for projectional editing. Extensions like the little DPL projects `dtree` (https://youtu.be/dZ4p7T3VTcg) and `sm` (https://youtu.be/ecJGkrpUhQQ) show that it is possible to map DPL syntax onto TPL languages, hence, we are in a position to extend PL "syntax" to DaS (Diagrams as Syntax).

The issue of tool gravity boils down to the overuse of the synchronous sequential paradigm. That paradigm is how CPUs work. If your problem is that of debugging low level machine code then standard debuggers are useful, but, if your problem consists of debugging some higher level semantic design then you need better debuggers.

This reframes the tooling gravity objection significantly. You're not just saying "little languages can have good tools too" — you're saying **the dominant tools are already broken for anything above the machine level**, so the tooling gravity argument partly evaporates. The gravity exists, but it's pulling toward a local minimum.

**The DI (Design Intent) point is the crux**

A debugger built on sync/seq assumptions forces you to reconstruct design intent from execution traces — you're reading tea leaves. If the language's notation _is_ the design intent (as a well-designed little language or DPL should be), then the gap between "what I meant" and "what I can observe" shrinks dramatically. Your probe-as-extra-diagram-part is a perfect illustration: the probe lives at the semantic level of the design, not at the level of memory addresses and call stacks.

This is also why printf debugging works better than its reputation in higher-level paradigms. Its weakness in C isn't the concept — it's that C's notation sits so close to the machine that print output doesn't map cleanly back to design-level concepts. In a Lisp REPL or a Smalltalk image, the same idea becomes genuinely powerful precisely because the notation reflects the semantics.

**The GT / Moldable Development connection**

What GT is doing is essentially acknowledging that _no single view is the right debugger_ — different objects, different problems, different stages of development need different lenses. This is the logical endpoint of your argument: if sync/seq debuggers are paradigm-locked, the solution isn't a better sync/seq debugger but a meta-tool for constructing _paradigm-appropriate_ views. GT's "moldable" approach is that meta-tool, applied to a single rich environment. Your little-languages approach distributes the same insight — each little language brings its own natural observation points.

**On DaS and projectional editors**

The standard projectional editor assumption — that there's one canonical AST with multiple syntax skins — is subtly limiting, as you note. It still privileges the textual/structural representation as ground truth, with diagrams as a rendering. What you're pointing toward with DaS is stronger: the diagram _is_ the syntax, not a view of it. The `dtree` and `sm` examples are interesting precisely because they don't treat the diagram as secondary — the diagrammatic form carries information (topology, concurrency, fan-out) that the textual form must laboriously reconstruct.

Common Lisp as a substrate for this is an underappreciated idea. Its homoiconicity and macro system mean the "bare normalized syntax" is genuinely malleable — you're not fighting a fixed grammar to introduce new notational forms. The normalized syntax is almost a factbase of triples by another name: s-expressions are already a flat, regular, machine-friendly representation that doesn't privilege any particular semantic interpretation.

**The deeper point**

You're essentially arguing that the entire debugger ecosystem is a consequence of a historical accident — CPUs are sync/seq, so the first languages were sync/seq, so the first debuggers were sync/seq, and now we have tooling gravity toward a paradigm that was never chosen for its expressive power. Breaking out requires not better tools within the paradigm but tools built around semantic observability at the design level — which is only possible if the language's notation reflects design intent in the first place.
