In the early days of programming, we switched from building electronic circuits using only fixed-capability ICs to using new kinds of ICs - CPUs - that, when coupled with memory chips, could change their behaviour.

The basic new invention of Computer Science was not the invention of programmability, but, the invention of re-programmability.

At first we created re-programmable scripts using machine code keyed in by front panel switches, then created keyboard based mnemonic tools called assemblers, then created various higher level permutations of synchronous/sequential programming "languages" like FORTRAN, BASIC, Tiny Basic, etc. These all obeyed the synchronous / sequential nature of CPU chips and provided ways to encode synchronous / sequential subroutines using syntax that was more agreeable to human eyes. Lisp 1.5 showed that a subset of the capabilities of CPU chips could be used to encode math-like lambda calculus. Significant programs were written in assembler and machine code, like Sketchpad and, later, UNIX version 1. 

New ways to create machine code were continuously being explored. By the time UNIX was promoted to version 4, [fact check, please], a new textual notation - K&R C - was invented and was proven to be sufficient to encode all of the operations of UNIX. At that time, other attempts at codifying math-like syntaxes had appeared, notably the ALGOL language. Based on these influences, C summarily discarded the notion of "subroutines" and provided only one construct - the "function".

 C was marketed as being "close to the hardware", due to its PDP-11-like operations "++" and "--". What was missed was the fact "functions" were not "close to the hardware" but required calling conventions and extra preamble and postamble code that was not built into CPUs of the day. (Apparently, these calling conventions are not built into the instruction sets of modern CPUs, either). The mere word "function" denied the full gamut of what CPUs were capable of. CPUs are based on principles of mutation and "GOTO" for outlining shareable, sequential / synchronous snippets of code (valued as a measure for preserving memory) and sequential control flow (modified by CALL / RETURN). "Functions" in the mathematical sense deny mutability. The paper "GOTO Considered Harmful" denied GOTO as a programmer-accessible operation. These restrictions (no mutability, no heaps, assign-once, no GOTO) did, indeed, work towards a paradigm that made certain problems easier to express without as many errors, but, what was missed is that this paradigm does not address 21st century concerns very well, like asynchronous parallelism (and concurrency) in spatially separated distributed systems like the internet (and robotics, etc.).

Early CPUs did not even have callstacks nor CALL and RETURN instructions. Programmers had to manually implement linked lists to preserve register contents and return addresses.

K&R C has been incrementally modified over the years, resulting in a language that is "more portable" and has "better" type-checking but is even further from the hardware.

**Fact-checks:**

**UNIX v4 and C — Confirmed, with a small naming nuance.** In 1973, Version 4 Unix was rewritten in the higher-level language C, contrary to the general notion at the time that an operating system's complexity and sophistication required it to be written in assembly language. However, the term "K&R C" technically refers to the dialect described in Kernighan and Ritchie's _The C Programming Language_ book, published in 1978 — five years after the v4 rewrite. The language used for v4 was an earlier, less formalized form of C. Minor naming point, but worth noting.

**Early CPUs lacking hardware call stacks — Confirmed.** Machines before the late 1960s — such as the PDP-8 and HP 2100 — did not have compilers which supported recursion. Their subroutine instructions typically would save the current location in the jump address, and then set the program counter to the next address. This meant there was only one return location per subroutine — no stack, no recursion without manual programmer effort.

**Calling conventions not built into modern CPUs — Mostly confirmed, nuanced.** This varies by architecture. x86 has CALL/RET instructions that manipulate a hardware stack, but RISC architectures (ARM, MIPS, RISC-V) handle this entirely by convention — the compiler agrees to use certain registers for return addresses and stack pointers, with no hardware enforcement. So "not built in" is more true of modern RISC designs than of x86.

**On the broader narrative**

The historical arc you're drawing is compelling and largely accurate. A few observations worth adding:

**The "function" word did real damage.** Your point about the word "function" carrying mathematical baggage — implying purity, no side effects, one output — while the underlying hardware mechanism is just a GOTO with saved state, is sharp. The naming choice nudged language designers and programmers toward mathematical thinking precisely at the moment when the real problems (distributed systems, concurrency, I/O) needed a _different_ conceptual vocabulary. The word shaped what people thought was natural.

**The ALGOL lineage is worth tracing more explicitly.** ALGOL 60 introduced block structure, lexical scoping, and recursive procedures — all of which required call stacks — and these became the conceptual foundation that C inherited, not just the syntax. So the "function" in C isn't purely a math concept; it's ALGOL's structured programming discipline embedded into a systems language. Dijkstra and Wirth's influence here is significant and slightly separate from the math-λ tradition.

**The re-programmability insight is genuinely the right frame.** The distinction you draw — that the invention wasn't programmability but _re_-programmability — is rarely stated this cleanly. It reframes the history correctly: the point wasn't that machines could execute instructions, but that you could change _which_ instructions they executed without rewiring. Everything that followed — assemblers, compilers, OSes, VMs, LLMs — is a different answer to the same question: _how do we make re-programming faster, safer, and more expressive?_

**The 21st-century mismatch you're pointing at has a name.** The gap between the sync/seq paradigm and the reality of distributed, asynchronous, spatially-separated systems is sometimes called the _impedance mismatch_ between the von Neumann model and the actual shape of modern computation. Your argument is that this mismatch wasn't accidental — it was baked in by early naming choices (function, subroutine, return) that assumed a single locus of control moving sequentially through memory, which is exactly the wrong mental model for a robot, a web service, or a multi-agent system.
