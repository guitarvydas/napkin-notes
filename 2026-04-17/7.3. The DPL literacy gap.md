The reframing of _why_ DPLs have struggled is the important correction here. The standard explanation is tooling — version control, diffing, text ecosystem gravity. Your explanation goes deeper: DPLs have been used for the wrong things, trying to express what TPLs already express well, instead of the things diagrams are _genuinely better at_ — topology, connectivity, concurrency, fan-out, black-box boundaries. A DPL that tries to encode sequential logic is fighting its own medium just as much as a TPL that tries to encode statecharts.

**The coexistence point dissolves a false dichotomy**

The debate is usually framed as "should we use diagrams or text?" — which is the wrong question, for the same reason "should we use a hammer or a screwdriver?" is the wrong question. The workflow question is: _which tool for which aspect?_ TPLs for math-like transformations, sequential logic, expressions. DPLs for topology, control flow, connectivity, system architecture. The hybrid workflow isn't a compromise — it's a more honest acknowledgment of what each medium is actually good for.

**The draw.io/Excalidraw observation is underappreciated**

The fact that modern diagram editors already serialize to structured text is a significant but overlooked development. It means the "diagrams can't be version-controlled" objection is largely already solved at the storage layer — the problem is just that nobody has built the right diff/merge tooling on top of it, because the assumption persists that diagrams are inherently unversioned artifacts. The gap is a tooling gap, not a fundamental one.

Your sorting observation is practical and probably sufficient for most version control needs. The core requirement for git-friendly diffs is _stability_ and _determinism_ in serialization — the same diagram should always produce the same text. Sorted JSON keys, sorted XML attributes, canonical element ordering — these are straightforward normalizations. The fact that the result is human-unreadable at the raw text level doesn't matter, because the diagram editor _is_ the human-readable layer. Nobody reads `.docx` XML directly either.

**The PEG/OhmJS point changes the economics**

The traditional equation was: parsing = compiler project = months of infrastructure work = only worth it for general-purpose languages. PEG technologies break that equation. If parsing a draw.io XML export into a factbase of triples takes days rather than months, then the cost of building a custom DPL toolchain drops below the threshold where it becomes practical for a single person or small team. This is a phase transition in what's possible, not just an incremental improvement. It means the hybrid workflow — diagram editors for topology, PEG parsers as the bridge, TPLs for computation — is now actually buildable without a large team or institutional backing.

**The remaining hard problem**

The one genuinely unsolved piece is _semantic_ diffing rather than textual diffing. Two versions of a diagram that represent meaningfully different control flow might produce very similar JSON, while a trivial layout change (moving a box without changing any connections) might produce a large textual diff. This is the same problem version control has with binary formats, and the solution is probably the same: a domain-aware diff tool that understands the semantics of the diagram format — what constitutes a meaningful change versus a cosmetic one. A factbase representation actually helps here, because topology facts are separable from layout facts, and you can diff only the topology if that's what you care about.
