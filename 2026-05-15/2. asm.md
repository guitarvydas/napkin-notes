
write pseudo-asm code for 
`a := 1; b := 42; c := a + b;` 

```
load R1,#1 
load R2,#42 
add R0,R1,R2
```

- Move the constant `1` into register `R1` (I'm arbitrarily using right-to-left operand syntax, subject first, then value(s) - this is meant to be read "load R1 with the constant value 1")
- Move the constant `42` into register `R2`
- Add the contents of `R1` and `R2` and store the result in `R0`
- This version of assembler code assumes that `R1` is `a`, that `R2` is `b` and that `R0` is `c`. If the intent is to store these values into actual RAM memory locations, then we need more code.

or

```
load R1,#1 
add  R1,#42 
```

- This is a optimization of the above, where the result is left in `R1`
- Move the constant `1` into `R1`
- Add the constant `42` to `R1`
- Leave the result in `R1`

or

```
load R0,#1 
store A,R0
load R0,#42
store B,R0
load R0,A
add R0,B
store C,R0
halt
A: alloc 1
B: alloc 1
C: alloc 1
```

- Similar to above, except that we ensure that the values are stored in RAM cells call "A", "B" and "C".
- the memory cells "A", "B" and "C" are explicitly defined and allocated after the "halt" instruction.

---

- If we invent opcode operand modes that allow us to use RAM cells as well as registers, we might get fewer lines of assembler code, something like:
```
load (A),#1
load (B),#42
add (C),(A),(B)
halt
A: alloc 1
B: alloc 1
C: alloc 1
```

Assembler code maps 1:1 with machine code, so, fewer lines of assembler means fewer machine code instructions, which probably maps to faster execution.

The notation `(A)` means to access a memory cell indirectly through the address associated with the label "A". The instruction `load (A),#1` doesn't store `1` into a non-entity called `A`, but stores `1` into the memory cell labelled `A`. A memory slot is created with the `alloc 1` instruction and is given the human-readable text label "A" (just after the `halt` instruction). The assembly language assembler program calculates the actual index (address) of the memory slot labelled `A`.

then write 
```
function plus (a, b) { return a + b; } c := plus (1, 42);
```

```
plus:
  load R0,(FP+0)
  add R0,(FP+1)
  return

load R0,#42             ;; C calling convention, 1st arg at top of stack
push R0
load R0,#1
push R0
call plus
store C, R0
```

- The functional paradigm / model, requires us to devote one register to be the frame pointer (FP) and another register to be the stack pointer (SP)
- The syntax `(FP+...)` means: access the contents at RAM location pointed to by the frame pointer plus an offset.
- Here, we're assuming that all cells in RAM are the same size (1 byte or 2 bytes or 4 bytes, etc.), so `(FP+1)` means the RAM cell at the location indexed by the index contained in the FP register plus the offset (either as 1 byte or 2 bytes or 4 bytes)

```
function plus (a, b) { return a + b; }
function subtract (a, b) { return a - b; }

process "X" { c := plus (1, 42) }
process "Y" { c := subtract (2, 43) }
```

```
plus:
  load R0,(FP+0)
  add R0,(FP+1)
  return

subtract:
  load R0,(FP+0)
  subtract R0,(FP+1)
  return

processX:
  load R0,#42
  push R0
  load R0,#1
  push R0
  call plus
  store C, R0
  halt
  
processY:
  load R0,#43
  push R0
  load R0,#2
  push R0
  call subtract
  store C, R0
  halt
  
```

```
main:
  load SP,StackX
  push SP,&processX
  return
context_switch:
  ;; stack = PC,a,b
  push R0
  load R0,(nextPID)
  inc nextPID
  mod nextPIDlist,2     ;; wrap around PID index using modulo
nextPIDlist: 
  alloc &PIDX
  alooc &PIDY
StackX:
  alloc 1               ;; parameter a
  alloc 1               ;; parameter b
  alloc 1               ;; return address
StackY:
  alloc 1               ;; parameter a
  alloc 1               ;; parameter b
  alloc 1               ;; return address
PIDX:
  alloc 1               ;; SP
  alloc 1               ;; FP
  alloc 1               ;; PC
PIDY:
  alloc 1               ;; SP
  alloc 1               ;; FP
  alloc 1               ;; PC
```
